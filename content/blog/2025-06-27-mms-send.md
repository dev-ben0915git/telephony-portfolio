## 第1章 概述

### 1.1 MMS在Android Telephony架构中的位置

彩信（MMS，Multimedia Messaging Service）的发送在Android系统中横跨多个进程和子系统，其架构复杂度远超电路域短信（SMS）。从进程视角来看，一次完整的MMS发送涉及三个核心进程的协作：

| 进程 | 核心组件 | 职责 |
|------|---------|------|
| 发送方应用进程（如AOSP Messaging） | `SmsManager.sendMultimediaMessage` | 构造MMS PDU、发起跨进程调用 |
| 系统服务进程（system_server） | `IMms` AIDL接口 → `MmsService` | 接收请求、管理队列、调度网络、执行HTTP POST |
| Telephony进程（com.android.phone） | `DataNetworkController` → `DataNetwork` → `TelephonyNetworkAgent` | 建立MMS专用数据连接、通告网络给ConnectivityManager |

与SMS的根本差异在于：SMS通过RIL直接发送AT指令走信令通道（CS域），而MMS本质上是一次HTTP POST请求，必须依赖数据连接（PS域）访问MMSC（Multimedia Messaging Service Centre）。这意味着MMS发送前需要经历完整的"网络请求 → 数据链路建立 → 网络通告"流程，任何环节的失败都会导致MMS发送失败。

跨进程调用链如下：

```
Messaging App (SmsManager.sendMultimediaMessage)
  └─→ IMms.sendMessage (AIDL, system_server进程)
        └─→ MmsService.sendMessageWithoutUpgrade
              └─→ SendRequest.addSimRequest → execute
                    ├─ MmsNetworkManager.acquireNetwork
                    │    └─ ConnectivityManager.requestNetwork (NET_CAPABILITY_MMS)
                    │         └─ TelephonyNetworkProvider → DataNetworkController
                    │              └─ DataNetwork (状态机) → TelephonyNetworkAgent
                    ├─ ApnSettings.load (获取MMSC URL)
                    └─ MmsHttpClient.execute (HTTP POST 到MMSC)
```

### 1.2 整体流程总览

MMS发送从应用层调用到最终结果返回，共经历以下10个关键步骤：

| 步骤 | 阶段 | 核心类/方法 | 说明 |
|------|------|------------|------|
| 1 | 请求发起 | `SmsManager.sendMultimediaMessage` | 应用通过SmsManager API发起MMS发送，传入PDU Uri、目标subId和PendingIntent |
| 2 | 跨进程传输 | `IMms.sendMessage` (AIDL) | 通过AIDL接口将请求从应用进程传递到system_server中的MmsService |
| 3 | 预校验 | `MmsService.sendMessageWithoutUpgrade` | 校验subId合法性、活跃状态、MMS配置加载、MMS开关状态、数据连接可用性 |
| 4 | 请求入队 | `MmsService.addSimRequest` → `addToRunningRequestQueueSynchronized` | 按SIM隔离策略将请求加入发送队列（`QUEUE_INDEX_SEND`），由线程池调度 |
| 5 | PDU读取 | `SendRequest.prepareForHttpRequest` → `readPduFromContentUri` | 从ContentProvider Uri读取PDU二进制数据，最大不超过`MMS_CONFIG_MAX_MESSAGE_SIZE` |
| 6 | 网络获取 | `MmsNetworkManager.acquireNetwork` | 构造`NetworkRequest`（`NET_CAPABILITY_MMS` + `TRANSPORT_CELLULAR`），向ConnectivityManager请求MMS网络 |
| 7 | 数据链路建立 | `DataNetworkController.onSatisfyNetworkRequest` → `DataNetwork.ConnectingState` | Telephony层评估请求、匹配DataProfile、建立PS数据连接、创建TelephonyNetworkAgent通告网络 |
| 8 | APN加载 | `ApnSettings.load` | 从Telephony.Carriers数据库加载MMS APN配置（MMSC URL、代理地址、端口） |
| 9 | HTTP发送 | `MmsHttpClient.execute` (POST) | 通过获取到的MMS专用Network向MMSC发送HTTP POST请求，PDU作为请求体 |
| 10 | 结果处理 | `SendRequest.processResult` | 解析SendConf响应PDU，持久化消息到数据库，通过PendingIntent返回结果 |

### 1.3 MMS与SMS架构对比

| 维度 | SMS（CS短信） | MMS（彩信） |
|------|--------------|-------------|
| **传输通道** | RIL接口 → 基带信令（CS域） | HTTP POST → MMSC（PS域数据连接） |
| **数据链路依赖** | 无需数据连接，仅依赖CS注册 | 必须建立MMS专用数据连接（`NET_CAPABILITY_MMS`） |
| **PDU格式** | 3GPP TS 23.040定义的SMS PDU（TPDU） | OMA-TS-MMS-enc定义的MMS PDU（Binary Encoded） |
| **响应格式** | RIL返回发送结果status code | MMSC返回HTTP 200 + SendConf PDU |
| **服务端** | SMSC（短信中心）通过信令接收 | MMSC（彩信中心）通过HTTP接收 |
| **请求大小** | 单条SMS最大160字节（7-bit编码） | 由`MMS_CONFIG_MAX_MESSAGE_SIZE`配置，默认通常为300KB-1MB |
| **进程架构** | SmsManager → ISms/SmsDispatcher → RIL | SmsManager → IMms/MmsService → MmsHttpClient → MMSC |
| **重试机制** | 由RIL/Modem层处理 | MmsRequest内置3次重试，含指数退避（2s → 4s → 8s） |
| **并发控制** | 可并发发送多条SMS | 同一时刻只允许一个SIM的MMS请求在执行（`mCurrentSubId`互斥） |

最关键的区别在于**数据链路依赖**：SMS发送不需要任何数据网络，而MMS发送必须先获得一个具有`NET_CAPABILITY_MMS`能力的网络连接。这意味着在飞行模式、无数据套餐、数据漫游关闭等场景下，MMS将无法发送。

### 1.4 关键术语解释

| 术语 | 全称 | 说明 |
|------|------|------|
| **MMSC** | Multimedia Messaging Service Centre | 彩信服务中心，运营商部署的HTTP服务器，负责接收、存储和转发MMS消息。MMS发送时通过HTTP POST将PDU提交到MMSC，下载时通过HTTP GET从MMSC获取 |
| **SendReq / SendConf** | Send Request / Send Confirm | OMA MMS规范中定义的两种PDU类型。`SendReq`是发送方向的请求PDU（对应发送操作），`SendConf`是MMSC返回的确认响应PDU（包含Response Status和Message ID） |
| **MMS APN** | MMS Access Point Name | 运营商为MMS流量配置的专用接入点。与互联网APN分离，通常包含MMSC URL、代理地址和代理端口等关键参数 |
| **NET_CAPABILITY_MMS** | Network Capability MMS | Android网络能力标志，标识一个网络连接具有承载MMS流量的能力。MmsNetworkManager通过此标志向ConnectivityManager请求专用网络 |
| **PduComposer / PduParser** | - | OMA MMS PDU的序列化与反序列化工具类。`PduComposer`将Java对象形式的PDU编码为二进制字节流（用于发送），`PduParser`将二进制字节流解析为Java对象（用于接收/解析响应） |

## 第2章 关键类与数据结构

### 2.1 SmsManager.sendMultimediaMessage

`SmsManager`是Android面向应用层的短信/彩信管理入口类。发送彩信的方法签名为：

```java
// android.telephony.SmsManager
public void sendMultimediaMessage(
    Context context,
    Uri contentUri,          // MMS PDU的ContentProvider Uri
    String locationUrl,      // 可选：目标MMSC URL覆盖
    Bundle configOverrides,  // 可选：MMS配置覆盖项
    PendingIntent sentIntent // 发送结果回调
)
```

该方法内部调用流程：

1. 获取`IMms`服务接口的Binder代理
2. 调用`IMms.sendMessage(subId, callingUser, callingPkg, contentUri, locationUrl, configOverrides, sentIntent, messageId, attributionTag)`
3. 应用进程通过ContentProvider Uri暴露MMS PDU数据，MmsService通过`ContentResolver.openFileDescriptor`跨进程读取

`contentUri`通常是应用私有ContentProvider生成的临时Uri，通过`FLAG_GRANT_READ_URI_PERMISSION`授权给系统进程读取。PDU数据在此Uri中以二进制流的形式存储，格式为OMA MMS Binary Encoded PDU。

### 2.2 IMms AIDL接口

`IMms`是定义在`com.android.internal.telephony`包下的AIDL接口，文件路径为`base/mms/java/com/android/internal/telephony/IMms.aidl`。该接口是应用进程与MmsService之间的跨进程通信契约。完整接口方法列表：

```java
interface IMms {
    // ===== MMS 发送/下载 =====
    void sendMessage(int subId, in int callingUser, String callingPkg,
            in Uri contentUri, String locationUrl, in Bundle configOverrides,
            in PendingIntent sentIntent, in long messageId, String attributionTag);

    void downloadMessage(int subId, in int callingUser, String callingPkg,
            String locationUrl, in Uri contentUri, in Bundle configOverrides,
            in PendingIntent downloadedIntent, in long messageId, String attributionTag);

    // ===== 消息导入/删除/更新 =====
    Uri importTextMessage(String callingPkg, String address, int type,
            String text, long timestampMillis, boolean seen, boolean read);
    Uri importMultimediaMessage(in int callingUser, String callingPkg,
            in Uri contentUri, String messageId, long timestampSecs, boolean seen, boolean read);
    boolean deleteStoredMessage(String callingPkg, in Uri messageUri);
    boolean deleteStoredConversation(String callingPkg, long conversationId);
    boolean updateStoredMessageStatus(String callingPkg, in Uri messageUri,
            in ContentValues statusValues);
    boolean archiveStoredConversation(String callingPkg, long conversationId, boolean archived);

    // ===== 草稿 =====
    Uri addTextMessageDraft(String callingPkg, String address, String text);
    Uri addMultimediaMessageDraft(in int callingUser, String callingPkg, in Uri contentUri);

    // ===== 其他 =====
    void sendStoredMessage(int subId, String callingPkg, in Uri messageUri,
            in Bundle configOverrides, in PendingIntent sentIntent);
    void setAutoPersisting(String callingPkg, boolean enabled);
    boolean getAutoPersisting();
}
```

关键设计要点：
- 所有方法均有`enforceSystemUid()`权限检查，仅允许SYSTEM_UID调用
- `sendMessage`的`locationUrl`参数允许覆盖默认MMSC URL，这在某些运营商场景下是必需的
- `sendStoredMessage`方法当前抛出`UnsupportedOperationException`，表明该功能已废弃

### 2.3 MmsService

`MmsService`（包路径`com.android.mms.service`）是运行在system_server进程中的Android Service，实现了`MmsRequest.RequestManager`接口，是MMS操作的核心调度中心。

**核心字段与数据结构：**

```java
public class MmsService extends Service implements MmsRequest.RequestManager {
    // 队列索引常量
    public static final int QUEUE_INDEX_SEND = 0;     // 发送队列
    public static final int QUEUE_INDEX_DOWNLOAD = 1;  // 下载队列

    // 线程池：每个队列固定4个线程
    public static final int THREAD_POOL_SIZE = 4;
    private final ExecutorService[] mRunningRequestExecutors = new ExecutorService[2];

    // 当前正在运行的请求使用的SIM ID（同一时刻只有一个SIM的请求在执行）
    private int mCurrentSubId;
    // 当前正在运行的请求计数
    private int mRunningRequestCount;

    // 按SIM等待的请求队列（不同SIM的请求需要串行化）
    private final Queue<MmsRequest> mPendingSimRequestQueue = new ArrayDeque<>();

    // 每个SIM的MmsNetworkManager缓存
    private final SparseArray<MmsNetworkManager> mNetworkManagerCache = new SparseArray<>();

    // PDU传输线程池（用于从ContentProvider读取/写入PDU）
    private final ExecutorService mPduTransferExecutor = Executors.newCachedThreadPool();

    // AIDL Stub实现
    private IMms.Stub mStub = new IMms.Stub() { ... };
}
```

**SIM互斥调度机制：**

MmsService实现了一种严格的SIM互斥策略——同一时刻只允许一个SIM的MMS请求处于执行状态。这是通过`mCurrentSubId`和`mPendingSimRequestQueue`协作实现的：

```java
@Override
public void addSimRequest(MmsRequest request) {
    synchronized (this) {
        if (mPendingSimRequestQueue.size() > 0 ||
                (mRunningRequestCount > 0 && request.getSubId() != mCurrentSubId)) {
            // 当前有其他SIM的请求在执行 → 加入等待队列
            mPendingSimRequestQueue.add(request);
        } else {
            // 可以直接执行
            addToRunningRequestQueueSynchronized(request);
        }
    }
}
```

当一组请求执行完毕后，`movePendingSimRequestsToRunningSynchronized`会从等待队列中取出同一SIM的后续请求继续执行，遇到不同SIM时停止。

### 2.4 MmsRequest（6种状态）

`MmsRequest`（包路径`com.android.mms.service`）是所有MMS操作的基类，定义了请求的生命周期状态机和通用的执行/重试逻辑。

**6种请求状态（`MmsRequestState`枚举）：**

| 状态 | 含义 | 进入条件 |
|------|------|---------|
| `Unknown` | 初始状态（构造后但实际上直接跳到Created） | 构造函数默认值（实际会被构造函数覆盖） |
| `Created` | 请求已创建 | 构造函数中设置 |
| `PrepareForHttpRequest` | 准备HTTP请求（读取PDU等） | `execute()`方法入口 |
| `AcquiringNetwork` | 正在获取MMS网络 | `networkManager.acquireNetwork()`调用前 |
| `LoadingApn` | 正在加载APN设置 | 网络获取成功后 |
| `DoingHttp` | 正在执行HTTP请求 | APN加载成功后 |
| `Success` | 请求成功完成 | `processResult`中result为OK时设置 |
| `Failure` | 请求失败 | `processResult`中result非OK时设置 |

实际上枚举定义了8个值，其中`Unknown`是默认值但被构造函数立即覆盖为`Created`，核心流转路径为：`Created → PrepareForHttpRequest → AcquiringNetwork → LoadingApn → DoingHttp → Success/Failure`。

**核心`execute`方法（包含重试逻辑）：**

```java
public void execute(Context context, MmsNetworkManager networkManager) {
    result = SmsManager.MMS_ERROR_UNSPECIFIED;
    byte[] response = null;
    int retryId = 0;
    currentState = MmsRequestState.PrepareForHttpRequest;

    if (!prepareForHttpRequest()) {       // 步骤1：准备PDU数据
        result = SmsManager.MMS_ERROR_IO_ERROR;
    } else {
        long retryDelaySecs = 2;
        for (retryId = 0; retryId < RETRY_TIMES; retryId++) {  // 最多重试3次
            try {
                listenToDataConnectionState(connectionStateCallback);
                currentState = MmsRequestState.AcquiringNetwork;
                int networkId = networkManager.acquireNetwork(requestId);  // 步骤2：获取网络
                currentState = MmsRequestState.LoadingApn;
                ApnSettings apn = loadApnSettings();                       // 步骤3：加载APN
                currentState = MmsRequestState.DoingHttp;
                response = doHttp(context, networkManager, apn);            // 步骤4：HTTP请求
                result = Activity.RESULT_OK;
                break;
            } catch (ApnException e) {
                result = SmsManager.MMS_ERROR_INVALID_APN;
                break;           // APN错误不重试
            } catch (MmsNetworkException e) {
                result = SmsManager.MMS_ERROR_UNABLE_CONNECT_MMS;
                break;           // 网络错误不重试
            } catch (MmsHttpException e) {
                result = SmsManager.MMS_ERROR_HTTP_FAILURE;
                // HTTP错误会重试（指数退避）
            } finally {
                networkManager.releaseNetwork(requestId,
                        this instanceof DownloadRequest && result == Activity.RESULT_OK);
                stopListeningToDataConnectionState(connectionStateCallback);
            }
            // 退避等待（2s → 4s → 8s）
            new CountDownLatch(1).await(retryDelaySecs, TimeUnit.SECONDS);
            retryDelaySecs <<= 1;
        }
    }
    processResult(context, result, response, httpStatusCode, false, retryId);
}
```

**抽象方法（由子类实现）：**

| 方法 | SendRequest实现 | DownloadRequest实现 |
|------|----------------|-------------------|
| `doHttp` | POST方式发送PDU到MMSC | GET方式从MMSC下载PDU |
| `getPendingIntent` | 返回`mSentIntent` | 返回`mDownloadedIntent` |
| `getQueueType` | `QUEUE_INDEX_SEND` (0) | `QUEUE_INDEX_DOWNLOAD` (1) |
| `persistIfRequired` | 持久化到Telephony.Mms.Sent | 持久化到Telephony.Mms.Inbox |
| `prepareForHttpRequest` | 从Uri读取PDU | 读取待下载的locationUrl |
| `transferResponse` | 将SendConf放入Intent extras | 将下载PDU写入ContentProvider |

**MonitorTelephonyCallback**：`MmsRequest`内部类，用于监听数据连接状态变化。通过`onPreciseDataConnectionStateChanged`回调跟踪当前MMS网络的APN信息和最后连接失败原因码，供`processResult`时通过PendingIntent的extra返回给调用方。

### 2.5 SendRequest

`SendRequest`（包路径`com.android.mms.service`）继承自`MmsRequest`，专门处理MMS发送操作。

**构造函数：**

```java
public SendRequest(RequestManager manager, int subId, Uri contentUri, String locationUrl,
        PendingIntent sentIntent, int callingUser, String creator,
        Bundle configOverrides, Context context, long messageId, MmsStats mmsStats,
        TelephonyManager telephonyManager)
```

**关键字段：**

```java
private final Uri mPduUri;          // PDU数据来源的ContentProvider Uri
public byte[] mPduData;             // 缓存的PDU二进制数据（从Uri读取后缓存）
private final String mLocationUrl;  // 可选的MMSC URL覆盖
private final PendingIntent mSentIntent;  // 发送结果回调
private final int mCallingUser;     // 调用者User ID
```

**doHttp方法实现（HTTP POST发送PDU）：**

```java
@Override
protected byte[] doHttp(Context context, MmsNetworkManager netMgr, ApnSettings apn)
        throws MmsHttpException {
    final MmsHttpClient mmsHttpClient = netMgr.getOrCreateHttpClient();
    final GenericPdu parsedPdu = parsePdu();
    notifyIfEmergencyContactNoThrow(parsedPdu);
    updateDestinationAddress(parsedPdu);
    return mmsHttpClient.execute(
            mLocationUrl != null ? mLocationUrl : apn.getMmscUrl(),  // 目标URL
            mPduData,                                                // 请求体
            MmsHttpClient.METHOD_POST,                               // HTTP方法
            apn.isProxySet(), apn.getProxyAddress(), apn.getProxyPort(),
            mMmsConfig, mSubId, requestId);
}
```

发送前处理流程：
1. `parsePdu()`：使用`PduParser`解析PDU数据，提取收件人等信息
2. `notifyIfEmergencyContactNoThrow()`：检查收件人是否为紧急号码，若是则通知`BlockedNumberContract`临时解除号码拦截
3. `updateDestinationAddress()`：调用`SmsNumberUtils.filterDestAddr`更新目标地址（VZW辅助拨号需求），如有变更则重新序列化PDU

**prepareForHttpRequest实现：**

```java
@Override
protected boolean prepareForHttpRequest() {
    return readPduFromContentUri();
}

private boolean readPduFromContentUri() {
    if (mPduData != null) return true;  // 已缓存
    final int bytesTobeRead = mMmsConfig.getInt(SmsManager.MMS_CONFIG_MAX_MESSAGE_SIZE);
    mPduData = mRequestManager.readPduFromContentUri(mPduUri, bytesTobeRead, mCallingUser);
    return (mPduData != null);
}
```

**Carrier Messaging Service发送路径：**

`SendRequest`支持通过运营商应用（`CarrierMessagingService`）发送MMS。如果系统检测到有运营商注册的`CarrierMessagingService`，会优先尝试通过运营商应用发送：

```java
public void trySendingByCarrierApp(Context context, String carrierMessagingServicePackage) {
    final CarrierSendManager carrierSendManger = new CarrierSendManager();
    final CarrierSendCompleteCallback sendCallback = new CarrierSendCompleteCallback(
            context, carrierSendManger);
    carrierSendManger.sendMms(context, carrierMessagingServicePackage, sendCallback);
}
```

回调处理中，如果运营商应用返回`SEND_STATUS_RETRY_ON_CARRIER_NETWORK`，则通过`maybeFallbackToRegularDelivery`回退到平台标准发送路径。

**persistIfRequired实现：**

发送完成后，如果调用者不是默认短信应用（`!SmsApplication.shouldWriteMessageForPackageAsUser`），系统需要自动将消息持久化到TelephonyProvider。`SendRequest`会将消息插入`Telephony.Mms.Sent.CONTENT_URI`，如果发送失败则更新`MESSAGE_BOX`为`MESSAGE_BOX_FAILED`，并解析`SendConf`响应提取`Response Status`和`Message ID`。

### 2.6 MmsNetworkManager

`MmsNetworkManager`（包路径`com.android.mms.service`）负责管理MMS专用的网络连接获取与释放。

**核心字段：**

```java
public class MmsNetworkManager {
    private Network mNetwork;                // 当前持有的MMS Network对象
    private int mMmsRequestCount;            // 使用该网络的MMS请求数
    private final NetworkRequest mNetworkRequest;  // 预构造的NetworkRequest
    private ConnectivityManager.NetworkCallback mNetworkCallback;
    private MmsHttpClient mMmsHttpClient;    // 当前网络对应的HTTP客户端
    private final int mSubId;                // 关联的订阅ID
    private boolean mIsSatelliteTransport;   // 是否为卫星传输
}
```

**NetworkRequest构造（关键）：**

```java
public MmsNetworkManager(Context context, int subId, Dependencies dependencies) {
    NetworkRequest.Builder builder = new NetworkRequest.Builder()
            .addTransportType(NetworkCapabilities.TRANSPORT_CELLULAR)
            .addCapability(NetworkCapabilities.NET_CAPABILITY_MMS)
            .setNetworkSpecifier(new TelephonyNetworkSpecifier.Builder()
                    .setSubscriptionId(mSubId).build());

    // 支持卫星网络传输MMS
    builder.removeCapability(NetworkCapabilities.NET_CAPABILITY_NOT_RESTRICTED);
    builder.addTransportType(NetworkCapabilities.TRANSPORT_SATELLITE);
    builder.removeCapability(NetworkCapabilities.NET_CAPABILITY_NOT_BANDWIDTH_CONSTRAINED);

    mNetworkRequest = builder.build();

```

该NetworkRequest明确了三个关键属性：
- **Transport**：`TRANSPORT_CELLULAR` + `TRANSPORT_SATELLITE`（支持卫星回传）
- **Capability**：`NET_CAPABILITY_MMS`（MMS专用）
- **Specifier**：`TelephonyNetworkSpecifier`指定了目标subId
- 移除了`NOT_RESTRICTED`和`NOT_BANDWIDTH_CONSTRAINED`能力标志，以匹配卫星网络的特性

**acquireNetwork方法（获取网络）：**

```java
public int acquireNetwork(final String requestId) throws MmsNetworkException {
    int networkRequestTimeoutMillis = mDeps.getNetworkRequestTimeoutMillis();  // 默认30分钟
    synchronized (this) {
        mReleaseHandler.removeCallbacks(mNetworkReleaseTask);
        mMmsRequestCount += 1;

        if (mNetwork != null) {
            // 网络已可用，直接返回
            return mNetwork.getNetId();
        }

        // 注册SIM卡状态变化监听器
        if (!mSimCardStateChangedReceiverRegistered) {
            mPhoneId = mDeps.getPhoneId(mSubId);
            mContext.registerReceiver(mSimCardStateChangedReceiver,
                    new IntentFilter(TelephonyManager.ACTION_SIM_CARD_STATE_CHANGED));
            mSimCardStateChangedReceiverRegistered = true;
        }

        // 首次请求则发起新的NetworkRequest
        if (mNetworkCallback == null) {
            startNewNetworkRequestLocked(networkRequestTimeoutMillis);
        }

        // 阻塞等待网络可用（最长约30分5秒）
        try {
            this.wait(networkRequestTimeoutMillis
                    + mDeps.getAdditionalNetworkAcquireTimeoutMillis());
        } catch (InterruptedException e) { }

        // 注销SIM卡状态监听器
        if (mSimCardStateChangedReceiverRegistered) {
            mContext.unregisterReceiver(mSimCardStateChangedReceiver);
            mSimCardStateChangedReceiverRegistered = false;
        }

        if (mNetwork != null) {
            return mNetwork.getNetId();  // 成功
        }

        // 超时处理
        if (mNetworkCallback != null) {
            releaseRequestLocked(mNetworkCallback);
            this.notifyAll();
        }
        throw new MmsNetworkException("Acquiring network failed");
    }
}
```

**NetworkRequestCallback内部类（网络状态回调）：**

```java
private class NetworkRequestCallback extends ConnectivityManager.NetworkCallback {
    @Override
    public void onCapabilitiesChanged(Network network, NetworkCapabilities nc) {
        synchronized (MmsNetworkManager.this) {
            final boolean isAvailable =
                    nc.hasCapability(NetworkCapabilities.NET_CAPABILITY_NOT_SUSPENDED);
            if (network.equals(mNetwork) && !isAvailable) {
                mNetwork = null;
                mMmsHttpClient = null;
                return;
            }
            if (mNetwork == null && isAvailable) {
                mIsSatelliteTransport = nc.hasTransport(
                        NetworkCapabilities.TRANSPORT_SATELLITE);
                mNetwork = network;
                MmsNetworkManager.this.notifyAll();  // 唤醒等待线程
            }
        }
    }

    @Override
    public void onUnavailable() {
        synchronized (MmsNetworkManager.this) {
            releaseRequestLocked(this);
            MmsNetworkManager.this.notifyAll();
        }
    }

    @Override
    public void onLost(Network network) {
        synchronized (MmsNetworkManager.this) {
            if (network.equals(mNetwork)) {
                mNetwork = null;
                mMmsHttpClient = null;
            }
        }
    }
}
```

**releaseNetwork方法（释放网络）：**

```java
public void releaseNetwork(final String requestId, final boolean shouldDelayRelease) {
    synchronized (this) {
        if (mMmsRequestCount > 0) {
            mMmsRequestCount -= 1;
            if (mMmsRequestCount < 1) {
                if (shouldDelayRelease) {
                    // 延迟释放（用于DownloadRequest成功后发送Ack）
                    mReleaseHandler.removeCallbacks(mNetworkReleaseTask);
                    mReleaseHandler.postDelayed(mNetworkReleaseTask,
                            mNetworkReleaseTimeoutMillis);
                } else {
                    releaseRequestLocked(mNetworkCallback);
                }
            }
        }
    }
}
```

`shouldDelayRelease`参数在发送场景下为`false`（立即释放），在下载场景且成功时为`true`（延迟释放以便在同一网络上发送确认）。

**getOrCreateHttpClient方法：**

```java
public MmsHttpClient getOrCreateHttpClient() {
    synchronized (this) {
        if (mMmsHttpClient == null && mNetwork != null) {
            mMmsHttpClient = new MmsHttpClient(mContext, mNetwork, mConnectivityManager);
        }
        return mMmsHttpClient;
    }
}
```

注意：`MmsHttpClient`与`Network`对象绑定，当网络丢失时（`onLost`回调）`mMmsHttpClient`会被置为null，确保后续请求使用新的网络连接。

### 2.7 MmsHttpClient

`MmsHttpClient`（包路径`com.android.mms.service`）封装了MMS的HTTP通信细节，支持POST（发送）和GET（下载）两种方法。

**构造函数：**


**execute方法核心流程：**

```java
public byte[] execute(String urlString, byte[] pdu, String method,
        boolean isProxySet, String proxyHost, int proxyPort,
        Bundle mmsConfig, int subId, String requestId) throws MmsHttpException {
    // 1. 设置代理（如需要）
    Proxy proxy = Proxy.NO_PROXY;
    if (isProxySet) {
        proxy = new Proxy(Proxy.Type.HTTP,
                new InetSocketAddress(mNetwork.getByName(proxyHost), proxyPort));
    }

    // 2. IPv4可用性等待（IPv6-only网络场景）
    maybeWaitForIpv4(requestId, url);

    // 3. 打开连接
    connection = (HttpURLConnection) mNetwork.openConnection(url, proxy);
    connection.setConnectTimeout(mmsConfig.getInt(SmsManager.MMS_CONFIG_HTTP_SOCKET_TIMEOUT));
    connection.setReadTimeout(mmsConfig.getInt(SmsManager.MMS_CONFIG_HTTP_SOCKET_TIMEOUT));

    // 4. 设置公共HTTP头
    connection.setRequestProperty("Accept",
            "*/*, application/vnd.wap.mms-message, application/vnd.wap.sic");
    connection.setRequestProperty("Accept-Language", getCurrentAcceptLanguage(...));
    connection.setRequestProperty("User-Agent", userAgent);
    // x-wap-profile头（UAProf）
    if (!TextUtils.isEmpty(uaProfUrl)) {
        connection.setRequestProperty(uaProfUrlTagName, uaProfUrl);
    }
    // Connection: close（某些运营商要求）
    if (mmsConfig.getBoolean(KEY_MMS_CLOSE_CONNECTION_BOOL, false)) {
        connection.setRequestProperty("Connection", "close");
    }
    // 额外HTTP头（从mms_config.xml的httpParams解析）
    addExtraHeaders(connection, mmsConfig, subId);

    // 5. POST请求体（发送场景）
    if (METHOD_POST.equals(method)) {
        connection.setDoOutput(true);
        connection.setRequestMethod(METHOD_POST);
        connection.setRequestProperty("Content-Type",
                "application/vnd.wap.mms-message; charset=utf-8");
        connection.setFixedLengthStreamingMode(pdu.length);
        OutputStream out = new BufferedOutputStream(connection.getOutputStream());
        out.write(pdu);
        out.flush();
        out.close();
    }

    // 6. 读取响应
    int responseCode = connection.getResponseCode();
    if (responseCode / 100 != 2) {
        throw new MmsHttpException(responseCode, responseMessage);
    }
    InputStream in = new BufferedInputStream(connection.getInputStream());
    // 读取全部响应字节到ByteArrayOutputStream
    return byteOut.toByteArray();
}
```

**HTTP请求头汇总：**

| Header | Value | 来源 |
|--------|-------|------|
| `Accept` | `*/*, application/vnd.wap.mms-message, application/vnd.wap.sic` | 硬编码常量 |
| `Content-Type` (POST) | `application/vnd.wap.mms-message; charset=utf-8` 或 `application/vnd.wap.mms-message` | 由`MMS_CONFIG_SUPPORT_HTTP_CHARSET_HEADER`决定 |
| `User-Agent` | 运营商配置值 | `MMS_CONFIG_USER_AGENT` |
| `x-wap-profile` | UAProf URL | `MMS_CONFIG_UA_PROF_URL` |
| `Accept-Language` | 当前Locale + "en-US" | 动态生成 |
| `Connection` | `close` | 由`KEY_MMS_CLOSE_CONNECTION_BOOL`配置 |
| 额外Headers | `httpParams`配置项解析 | `MMS_CONFIG_HTTP_PARAMS` |

**宏替换机制：**

`addExtraHeaders`支持`httpParams`中的宏替换，格式为`##MACRO_NAME##`：

| 宏名称 | 替换值 |
|--------|-------|
| `LINE1` | `TelephonyManager.getLine1Number()` |
| `LINE1NOCOUNTRYCODE` | 去除国家码的电话号码 |
| `NAI` | Network Access Identifier（Base64编码），用于Sprint等运营商认证 |

```java
    while (cursor.moveToNext()) {
        // APN类型必须包含"mms"或为空（默认所有类型）
        if (isValidApnType(cursor.getString(COLUMN_TYPE), ApnSetting.TYPE_MMS_STRING)) {
            String mmscUrl = trimWithNullCheck(cursor.getString(COLUMN_MMSC));
            if (TextUtils.isEmpty(mmscUrl)) continue;  // MMSC URL不能为空

            // 验证MMSC URL格式
            new URI(mmscUrl);  // 无效则抛出ApnException

            // 解析代理信息

            int proxyPort = 80;  // 默认端口
            if (!TextUtils.isEmpty(proxyAddress)) {

                if (!TextUtils.isEmpty(portString)) {
                    proxyPort = Integer.parseInt(portString);
                }
            }
            return new ApnSettings(mmscUrl, proxyAddress, proxyPort, getDebugText(cursor));
        }
    }
    return null;
}
```

## 第3章 MMS网络请求与数据链路建立

### 3.1 MmsNetworkManager到ConnectivityManager的网络请求

MMS发送的第一步是获取具有`NET_CAPABILITY_MMS`能力的网络连接。该过程从`MmsNetworkManager`发起，经过ConnectivityManager，最终由Telephony层的`DataNetworkController`处理。

**请求发起入口：**

当`MmsRequest.execute`进入`AcquiringNetwork`状态时，调用`MmsNetworkManager.acquireNetwork(requestId)`。`acquireNetwork`内部调用`startNewNetworkRequestLocked`：

```java
private void startNewNetworkRequestLocked(int networkRequestTimeoutMillis) {
    final ConnectivityManager connectivityManager = getConnectivityManager();
    mNetworkCallback = new NetworkRequestCallback();
    connectivityManager.requestNetwork(
            mNetworkRequest,      // 预构造的NetworkRequest
            mNetworkCallback,     // 网络状态回调
            networkRequestTimeoutMillis);  // 超时时间（默认30分钟）
}
```

**NetworkRequest的构成：**

```java
NetworkRequest.Builder builder = new NetworkRequest.Builder()
        .addTransportType(NetworkCapabilities.TRANSPORT_CELLULAR)
        .addCapability(NetworkCapabilities.NET_CAPABILITY_MMS)
        .setNetworkSpecifier(new TelephonyNetworkSpecifier.Builder()
                .setSubscriptionId(mSubId).build());
// 卫星支持
builder.removeCapability(NetworkCapabilities.NET_CAPABILITY_NOT_RESTRICTED);
builder.addTransportType(NetworkCapabilities.TRANSPORT_SATELLITE);
builder.removeCapability(NetworkCapabilities.NET_CAPABILITY_NOT_BANDWIDTH_CONSTRAINED);
```

这个NetworkRequest向ConnectivityManager声明了：
- 需要蜂窝网络或卫星网络传输
- 必须具有MMS能力
- 关联到指定的SIM（通过`TelephonyNetworkSpecifier`）

**ConnectivityManager到TelephonyNetworkProvider的传递：**

ConnectivityManager收到`requestNetwork`调用后，会将NetworkRequest分发给所有注册的`NetworkProvider`。对于蜂窝网络，对应的Provider是`TelephonyNetworkProvider`（运行在com.android.phone进程）。TelephonyNetworkProvider根据请求中的`TelephonyNetworkSpecifier`确定目标SIM，然后将请求转发给该SIM对应的`DataNetworkController`。

**阻塞等待机制：**

`acquireNetwork`使用`Object.wait()`阻塞当前线程（即MmsService的发送线程池中的线程），直到`NetworkRequestCallback.onCapabilitiesChanged`中检测到网络可用后调用`notifyAll()`唤醒。这种设计意味着MMS发送线程在等待网络期间是阻塞的，不会释放线程资源，但由于线程池大小为4，一般不会造成资源耗尽。

**SIM卡状态快速失败机制：**

```java
// 在等待网络期间注册SIM卡状态监听器
mContext.registerReceiver(mSimCardStateChangedReceiver,
        new IntentFilter(TelephonyManager.ACTION_SIM_CARD_STATE_CHANGED));

// 监听器实现
private final BroadcastReceiver mSimCardStateChangedReceiver = new BroadcastReceiver() {
    @Override
    public void onReceive(Context context, Intent intent) {
        final int simState = intent.getIntExtra(TelephonyManager.EXTRA_SIM_STATE, ...);
        final int phoneId = intent.getIntExtra(PhoneConstants.PHONE_KEY, ...);
        if (mPhoneId == phoneId && simState == TelephonyManager.SIM_STATE_ABSENT) {
            // SIM卡被拔出 → 立即释放请求并唤醒等待线程
            releaseRequestLocked(mNetworkCallback);
            MmsNetworkManager.this.notifyAll();
        }
    };
};
```

这保证了在SIM卡被移除时，MMS请求能够快速失败而不是等到30分钟超时。

### 3.2 DataNetworkController评估MMS网络请求

`DataNetworkController`（包路径`com.android.internal.telephony.data`）是Telephony数据栈的核心模块，负责管理和创建所有移动数据网络。它是per-SIM的——DSDS设备上有两个实例。

**请求满足流程（`onSatisfyNetworkRequest`）：**

```java
private void onSatisfyNetworkRequest(@NonNull TelephonyNetworkRequest networkRequest) {
    // 1. 检查是否已满足
    if (networkRequest.getState() == TelephonyNetworkRequest.REQUEST_STATE_SATISFIED) {
        return;
    }

    // 2. 尝试附加到已有的兼容DataNetwork
    if (findCompatibleDataNetworkAndAttach(networkRequest)) {
        return;
    }

    // 3. 评估是否可以建立新网络
    DataEvaluation evaluation = evaluateNetworkRequest(networkRequest,
            DataEvaluationReason.NEW_REQUEST);
    if (!evaluation.containsDisallowedReasons()) {
        DataProfile dataProfile = evaluation.getCandidateDataProfile();
        if (dataProfile != null) {
            setupDataNetwork(dataProfile, null, evaluation.getDataAllowedReason());
        }
    }
}
```

**evaluateNetworkRequest评估逻辑（关键检查点）：**

对于MMS网络请求（`NET_CAPABILITY_MMS`），`evaluateNetworkRequest`执行一系列检查：

```java
public DataEvaluation evaluateNetworkRequest(
        @NonNull TelephonyNetworkRequest networkRequest, DataEvaluationReason reason) {
    DataEvaluation evaluation = new DataEvaluation(reason);

    // 1. 传输方式检查：是否能通过蜂窝/卫星传输满足
    int transport = mAccessNetworksManager.getPreferredTransportByNetworkCapability(
            networkRequest.getHighestPriorityApnTypeNetworkCapability());
    if (!canConnectivityTransportSatisfyNetworkRequest(networkRequest, transport)) {
        evaluation.addDataDisallowedReason(
                DataDisallowedReason.DATA_NETWORK_TRANSPORT_NOT_ALLOWED);
    }

    // 2. 不支持的NetworkCapability检查
    if (Arrays.stream(networkRequest.getCapabilities())
            .anyMatch(mDataConfigManager.getUnsupportedNetworkCapabilities()::contains)) {
        evaluation.addDataDisallowedReason(
                DataDisallowedReason.UNSUPPORTED_NETWORK_CAPABILITIES);
    }

    // 3. PS附着状态检查（非EIMS请求）
    if (!serviceStateAllowsPSAttach(mServiceState, transport)) {
        evaluation.addDataDisallowedReason(DataDisallowedReason.NOT_IN_SERVICE);
    }

    // 4. SIM状态检查
    if (mSimState != TelephonyManager.SIM_STATE_LOADED) {
        evaluation.addDataDisallowedReason(DataDisallowedReason.SIM_NOT_READY);
    }

    // 5. 运营商配置就绪检查
    if (!mDataConfigManager.isConfigCarrierSpecific()) {
        evaluation.addDataDisallowedReason(DataDisallowedReason.DATA_CONFIG_NOT_READY);
    }

    // 6. 数据漫游检查
    // 7. 数据开关检查
    // 8. 并发语音/数据检查
    // ...（更多检查项）

    networkRequest.setEvaluation(evaluation);
    return evaluation;
}
```

**评估结果处理：**

如果评估通过（不包含任何DisallowedReason），`DataNetworkController`调用`setupDataNetwork`创建新的`DataNetwork`实例。如果评估失败，请求保留在未满足池中，等待环境变化后重新评估。

## 第4章 MMS PDU构造与编解码

### 4.1 MMS PDU概述与OMA-TS-MMS-enc规范

MMS PDU（Protocol Data Unit）是彩信消息的二进制编码格式，由OMA（Open Mobile Alliance）在**OMA-TS-MMS-enc-V1_3**规范中定义。与SMS的TPDU基于3GPP TS 23.040不同，MMS PDU使用独立的编码体系，采用紧凑的二进制编码以适应移动网络的传输效率要求。

**MMS PDU的整体结构分为两部分：**

1. **MMS Header**：使用OMA定义的Binary Encoded格式，包含消息类型、发送者、接收者、时间戳等元信息
2. **MMS Body**：由一个或多个`PduPart`组成，每个Part包含Content-Type和内容数据（如图片、音频、文本等）

**PDU类型体系（基于`X-Mms-Message-Type`字段）：**

| PDU类型 | 值 | 方向 | Java类 | 说明 |
|---------|---|------|--------|------|
| `m-send-req` | 0x80 | MO（发送） | `SendReq` | 发送请求 |
| `m-send-conf` | 0x81 | MT（响应） | `SendConf` | 发送确认 |
| `m-notification-ind` | 0x82 | MT（通知） | `NotificationInd` | MMS下载通知 |
| `m-retrieve-conf` | 0x84 | MT（下载） | `RetrieveConf` | 下载响应（包含完整MMS内容） |
| `m-acknowledge-ind` | 0x85 | MO（确认） | `AcknowledgeInd` | 下载确认 |
| `m-delivery-ind` | 0x86 | MT（通知） | `DeliveryInd` | 投递报告 |
| `m-read-rec-ind` | 0x87 | MO（报告） | `ReadRecInd` | 阅读报告 |

在发送场景中，应用构造`SendReq` PDU并序列化为二进制后传递给MmsService；MMSC收到后返回`SendConf` PDU作为确认。Java类位于`com.google.android.mms.pdu`包中（来自AOSP的`external/libphonenumber`或`frameworks/base`中的mms库）。

**PDU的Java类继承体系：**

```
GenericPdu (abstract)
  ├─ SendReq extends MultimediaMessagePdu
  ├─ SendConf extends GenericPdu
  ├─ RetrieveConf extends MultimediaMessagePdu
  ├─ NotificationInd extends MultimediaMessagePdu
  ├─ AcknowledgeInd extends GenericPdu
  ├─ DeliveryInd extends GenericPdu
  └─ ReadOrigInd extends GenericPdu

MultimediaMessagePdu extends GenericPdu
  └─ 包含PduBody（即PduPart集合）

PduHeaders
  └─ 包含所有MMS头字段

PduPart
  └─ 单个媒体part（Content-Type + 数据）
```

### 4.2 SendReq结构

`SendReq`（`com.google.android.mms.pdu.SendReq`）是MMS发送请求的PDU类型，继承自`MultimediaMessagePdu`。它包含了发送一条MMS消息所需的所有信息。

**SendReq包含的关键字段（通过PduHeaders常量索引）：**

| 字段名 | Header常量 | 类型 | 说明 |
|--------|-----------|------|------|
| `X-Mms-Message-Type` | `MESSAGE_TYPE` (0x0C) | Octet | 固定为`0x80`（m-send-req） |
| `X-Mms-MMS-Version` | `MMS_VERSION` (0x0D) | Octet | MMS协议版本（如`1.3` = `0x93`） |
| `To` | `TO` (0x97) | EncodedStringValue[] | 收件人地址列表 |
| `Cc` | `CC` (0x98) | EncodedStringValue[] | 抄送地址列表 |
| `Bcc` | `BCC` (0x99) | EncodedStringValue[] | 密送地址列表 |
| `From` | `FROM` (0x89) | EncodedStringValue | 发件人地址 |
| `Subject` | `SUBJECT` (0x96) | EncodedStringValue | 主题 |
| `X-Mms-Message-Class` | `MESSAGE_CLASS` (0x8A) | Byte/Text | 消息类别（Personal/Advertisement/Informational/Auto） |
| `X-Mms-Expiry` | `EXPIRY` (0x89) | Long | 消息过期时间（相对值） |
| `X-Mms-Priority` | `PRIORITY` (0x85) | Byte | 优先级（Low/Normal/High） |
| `X-Mms-Delivery-Report` | `DELIVERY_REPORT` (0x86) | Byte | 是否请求投递报告 |
| `X-Mms-Read-Reply` | `READ_REPLY` (0x87) | Byte | 是否请求阅读报告 |
| `Content-Type` | `CONTENT_TYPE` (0x04) | Octet/Text | MMS Body的整体Content-Type |
| `X-Mms-Transaction-Id` | `TRANSACTION_ID` (0x98) | Text | 事务ID（唯一标识） |
| `Date` | `DATE` (0x05) | Long | 发送时间 |

**源码中的使用示例（SendRequest.doHttp）：**

```java

            mMmsConfig.getBoolean(SmsManager.MMS_CONFIG_SUPPORT_MMS_CONTENT_DISPOSITION);

// 检查是否为紧急号码
private void notifyIfEmergencyContact(final GenericPdu parsedPdu) {
    if (parsedPdu != null && parsedPdu.getMessageType() == PduHeaders.MESSAGE_TYPE_SEND_REQ) {
        SendReq sendReq = (SendReq) parsedPdu;
        for (EncodedStringValue encodedStringValue : sendReq.getTo()) {
            if (isEmergencyNumber(encodedStringValue.getString())) {
                BlockedNumberContract.SystemContract.notifyEmergencyContact(mContext);
                return;
            }
        }
    }
}

// 更新目标地址（VZW辅助拨号）
private void updateDestinationAddress(final GenericPdu pdu) {

        boolean isUpdated = updateDestinationAddressPerType((SendReq)pdu, PduHeaders.TO);
        isUpdated = updateDestinationAddressPerType((SendReq)pdu, PduHeaders.CC) || isUpdated;
        isUpdated = updateDestinationAddressPerType((SendReq)pdu, PduHeaders.BCC) || isUpdated;
        if (isUpdated) {
```

### 4.3 PduHeaders

`PduHeaders`（`com.google.android.mms.pdu.PduHeaders`）是MMS PDU头的核心容器类，使用`SparseArray<Object>`存储所有头字段值。每个头字段由一个`short`类型的字段码索引。

**核心字段码常量（部分）：**

```java
// MMS PDU头字段码
public static final int BCC                          = 0x81;
public static final int CC                           = 0x82;
public static final int CONTENT_LOCATION            = 0x83;
public static final int CONTENT_TYPE                = 0x84;
public static final int DATE                         = 0x85;
public static final int DELIVERY_REPORT              = 0x86;
public static final int EXPIRY                       = 0x87;
public static final int FROM                         = 0x89;
public static final int MESSAGE_CLASS                = 0x8A;
public static final int MESSAGE_ID                   = 0x8B;
public static final int MESSAGE_TYPE                 = 0x8C;
public static final int MMS_VERSION                  = 0x8D;
public static final int PRIORITY                     = 0x85;
public static final int READ_REPLY                   = 0x86;
public static final int RESPONSE_STATUS              = 0x8C;

public static final int STATUS                       = 0x8A;
public static final int SUBJECT                      = 0x96;
public static final int TO                           = 0x97;
public static final int TRANSACTION_ID               = 0x98;
```

**字段值类型映射：**

根据OMA-TS-MMS-enc规范，不同字段使用不同的编码类型：

| 编码类型 | Java表示 | 适用字段示例 |
|---------|---------|------------|
| Octet（单字节） | `Integer` | MESSAGE_TYPE, MMS_VERSION, PRIORITY |
| Long Integer（变长） | `Long` | DATE, EXPIRY, CONTENT_SIZE |
| Text String | `byte[]` | TRANSACTION_ID, MESSAGE_ID |
| Encoded String Value | `EncodedStringValue` | FROM, TO, CC, BCC, SUBJECT |
| From（特殊格式） | `EncodedStringValue` | FROM（含Present-flag） |

**EncodedStringValue编码：**

地址字段（如To、From）使用`EncodedStringValue`封装，支持两种字符编码：
- **GSM 7-bit Default Alphabet**：`CHARSET_GSM_7BIT = 0x00`
- **UTF-8**：`CHARSET_UTF8 = 0x06`

地址值前缀含义：
- `+` 或数字开头：电话号码
- `info@domain.com`格式：Email地址

### 4.4 PduPart

`PduPart`（`com.google.android.mms.pdu.PduPart`）代表MMS消息体中的单个媒体part。一个MMS可以包含多个PduPart（如一张图片 + 一段文本）。

**PduPart的核心字段：**

| 字段 | Java类型 | 说明 |
|------|---------|------|
| `Content-Type` | `byte[]` | 该Part的MIME类型（如`image/jpeg`、`text/plain`） |
| `Content-Id` | `byte[]` | Part的唯一标识符（用于SMIL引用） |
| `Content-Location` | `byte[]` | Part的位置标识 |
| `Content-Disposition` | `byte[]` | Content-Disposition（如`attachment`、`inline`） |
| `Content-Transfer-Encoding` | `byte[]` | 内容传输编码（如`base64`、`binary`） |
| `Data` | `byte[]` | 实际媒体数据的字节数组 |

**典型MMS消息体结构：**

```
Content-Type: application/vnd.wap.multipart.related; start=<smil_id>; type="application/smil"
├─ Part 0: Content-Type: application/smil, Content-Id: <smil_id>, Data: SMIL presentation
├─ Part 1: Content-Type: image/jpeg, Content-Id: <image_id>, Data: JPEG图片数据
└─ Part 2: Content-Type: text/plain, Data: 文本内容
```

SMIL（Synchronized Multimedia Integration Language）part定义了MMS各媒体元素的展示顺序和布局。它是一个可选但广泛支持的标准part，使得MMS能够在不同设备上保持一致的展示效果。

**PduPart在源码中的使用：**

在`MmsService.readPduBytesFromContentUri`中，如果启用了`Flags.messagePromotion()`且PDU来自内部数据库Uri（`isInternalMmsUri`），会通过`PduPersister`加载PDU后用`PduComposer`重新序列化：


### 4.5 PduComposer序列化

`PduComposer`（`com.google.android.mms.pdu.PduComposer`）负责将Java对象形式的MMS PDU序列化为OMA MMS Binary Encoded格式的字节数组。这是发送MMS前PDU编码的核心工具类。

**序列化入口方法：**

```java
public byte[] make() {
    // 1. 序列化PDU头部
    ByteArrayOutputStream out = new ByteArrayOutputStream();
    encodeMessageType(mPdu);
    encodeMmsVersion(mPdu);
    // 根据PDU类型序列化各类头字段...
    // 2. 序列化PDU Body（如果有）
    if (mPdu instanceof MultimediaMessagePdu) {
        PduBody body = ((MultimediaMessagePdu) mPdu).getBody();
        encodeBody(body);
    }
    return out.toByteArray();
}
```

**头部编码规则（基于OMA-TS-MMS-enc规范）：**

每个头字段按照以下格式编码：
- **Text字段**：`[字段码] [长度] [字符数据]`
- **Octet字段**：`[字段码] [值]`
- **Long Integer字段**：`[字段码] [Short-length] [Long-integer-value]`
- **Encoded String Value字段**：`[字段码] [长度] [Charset] [String-value]`
- **From字段**：`[字段码] [Present-flag] [Address-value]`（Present-flag为1字节）

**Short-length编码：**

用于表示字符串或字节序列的长度，最高位为标志位：
- 若长度 < 128：单字节编码，最高位为0
- 若长度 >= 128：多字节编码，最高位为1，后跟一个或多个0x80 | (length >> N)字节

**Content-Type编码：**

Content-Type可以是Well-known值（单字节编码）或Text字符串：
- Well-known值：`0x23` = `application/vnd.wap.multipart.related`
- Text值：如 `image/jpeg`

**Multipart Body编码：**

```
[Headers Length] [Content-Type] [Multipart Body]
Multipart Body = Part1 + Part2 + ... + PartN
每个Part = [Headers Length] [Part Headers] [Part Data]
```

**源码中的序列化调用：**

### 4.6 PduParser反序列化

`PduParser`（`com.google.android.mms.pdu.PduParser`）负责将OMA MMS Binary Encoded格式的字节数组解析为Java对象形式的MMS PDU。它在以下场景被使用：

1. **发送前**：解析SendReq以提取收件人信息（`SendRequest.parsePdu`）
2. **接收响应后**：解析SendConf以提取Response Status（`SendRequest.persistIfRequired`）
3. **下载后**：解析RetrieveConf以提取MMS内容（DownloadRequest）
4. **导入MMS**：解析任意PDU以确定类型（`MmsService.parsePduForAnyCarrier`）

**解析入口方法：**

```java
public GenericPdu parse() {
    // 1. 解析Message-Type字段（必需，且必须是第一个字段）
    int messageType = parseUnsignedShort();  // 读取字段码
    if (messageType != PduHeaders.MESSAGE_TYPE) {
        throw new RuntimeException("Invalid MMS PDU, expected MESSAGE_TYPE");
    }
    byte typeValue = parseByte();  // 读取值
    mPduHeaders.setOctet(typeValue, messageType);

    // 2. 根据Message-Type创建对应的PDU对象
    switch (typeValue) {
        case PduHeaders.MESSAGE_TYPE_SEND_REQ:
            mPdu = new SendReq(); break;
        case PduHeaders.MESSAGE_TYPE_SEND_CONF:
            mPdu = new SendConf(); break;
        case PduHeaders.MESSAGE_TYPE_RETRIEVE_CONF:
            mPdu = new RetrieveConf(); break;
        // ... 其他类型
    }

    // 3. 循环解析后续头字段
    while (mPosition < mData.length) {
        int fieldCode = parseUnsignedShort();
        switch (fieldCode) {
            case PduHeaders.TO:
                mPduHeaders.setEncodedStringValue(parseEncodedStringValue(fieldCode), fieldCode);
                break;
            case PduHeaders.SUBJECT:
                mPduHeaders.setEncodedStringValue(parseEncodedStringValue(fieldCode), fieldCode);
                break;
            case PduHeaders.DATE:
                mPduHeaders.setLongInteger(parseLongInteger(fieldCode), fieldCode);
                break;
            // ... 其他字段
        }
    }

    // 4. 如果有Body，解析Multipart Body
    if (mPdu instanceof MultimediaMessagePdu) {
        parseBody((MultimediaMessagePdu) mPdu);
    }

    return mPdu;
}
```

**EncodedStringValue解析：**

```java
// 解析地址类型字段（To/Cc/Bcc/From/Subject等）
private EncodedStringValue parseEncodedStringValue(int fieldType) {
    int charsetIndex = parseCharset();  // 读取Charset标识
    byte[] text = parseWapString(charsetIndex);  // 按Charset解码字符串
    return new EncodedStringValue(charsetIndex, text);
}
```

**Long Integer解析：**

```java
// 解析时间戳、过期时间等Long类型字段
private long parseLongInteger(int startPos) {
    int length = parseValueLength();  // Short-length
    long value = 0;
    for (int i = 0; i < length; i++) {
        value = (value << 8) | (mData[mPosition++] & 0xFF);
    }
    return value;
}
```

**Content-Disposition兼容处理：**

`MmsService.parsePduForAnyCarrier`展示了一种关键的兼容性处理——某些运营商的MMS不正确地使用Content-Disposition字段，需要尝试两种解析方式：

```java
private static GenericPdu parsePduForAnyCarrier(final byte[] data) {
    GenericPdu pdu = null;
    try {
        // 先尝试启用Content-Disposition解析
        pdu = (new PduParser(data, true/*parseContentDisposition*/)).parse();
    } catch (RuntimeException e) { }
    if (pdu == null) {
        // 失败则回退到不解析Content-Disposition
        try {
            pdu = (new PduParser(data, false/*parseContentDisposition*/)).parse();
        } catch (RuntimeException e) { }
    }
    return pdu;
}
```

**isRawPduSendReq快速判断：**

`MmsService.sendMessage`中在判断发送失败是否需要弹出数据设置提示时，需要快速判断PDU类型是否为SendReq。为避免完整解析PDU，仅读取前2个字节：

```java
private boolean isRawPduSendReq(Uri contentUri, int callingUser) {
    byte[] pduData = new byte[2];
    int bytesRead = readPduBytesFromContentUri(contentUri, pduData, callingUser);
    // 第1字节必须是MESSAGE_TYPE字段码(0x0C)，第2字节必须是SEND_REQ值(0x80)
    if (bytesRead == 2
            && (pduData[0] & 0xFF) == MESSAGE_TYPE
            && (pduData[1] & 0xFF) == MESSAGE_TYPE_SEND_REQ) {
        return true;
    }
    return false;
}
```

这利用了OMA-TS-MMS-enc规范的编码特性：Message-Type总是PDU的第一个字段，且其字段码固定为`0x0C`（即`PduHeaders.MESSAGE_TYPE`），SendReq的类型值固定为`0x80`（即`PduHeaders.MESSAGE_TYPE_SEND_REQ`）。

## 第5章 MMS发送HTTP传输全流程

### 5.1 MmsHttpClient连接建立过程

`MmsHttpClient`（`com.android.mms.service.MmsHttpClient`）是MMS HTTP传输的核心类，负责建立到MMSC的HTTP连接并执行请求/响应的收发。它是`MmsNetworkManager`在成功获取MMS网络后创建的，每个`Network`实例对应一个`MmsHttpClient`。

**构造函数与Private DNS绕过：**

```java
// MmsHttpClient构造函数
public MmsHttpClient(Context context, Network network,
        ConnectivityManager connectivityManager) {
    mContext = context;
    // 关键：绕过Private DNS，因为MMSC位于运营商私有网络，
    // 第三方Private DNS服务器可能无法解析MMSC域名
    mNetwork = network.getPrivateDnsBypassingCopy();
    mConnectivityManager = connectivityManager;
}
```

`getPrivateDnsBypassingCopy()`是Android `Network`类提供的关键方法。MMSC通常使用运营商内部的私有域名或IP地址，如果用户配置了第三方Private DNS（如dns.google或1.1.1.1），这些DNS服务器无法解析运营商内部域名，会导致HTTP请求失败。此方法返回的Network副本绕过Private DNS，直接使用运营商DNS进行解析。

**MmsHttpClient的创建时机：**

`MmsNetworkManager.getOrCreateHttpClient()`采用懒创建+缓存策略：

```java
public MmsHttpClient getOrCreateHttpClient() {
    synchronized (this) {
        if (mMmsHttpClient == null) {
            if (mNetwork != null) {
                mMmsHttpClient = new MmsHttpClient(mContext, mNetwork, mConnectivityManager);
            }
        }
        return mMmsHttpClient;
    }
}
```

当`NetworkRequestCallback.onLost`或网络变为suspended时，`mMmsHttpClient`被置null，下次请求会重新创建，确保HTTP客户端始终绑定到可用的网络实例。

**直连模式 vs 代理模式：**

MMS HTTP连接支持两种模式，由APN配置中的代理设置决定：

| 维度 | 直连模式 | 代理模式 |
|------|---------|---------|
| **触发条件** | `apn.isProxySet()` 返回 false | `apn.isProxySet()` 返回 true |
| **代理对象** | `Proxy.NO_PROXY` | `new Proxy(Proxy.Type.HTTP, new InetSocketAddress(mNetwork.getByName(proxyHost), proxyPort))` |
| **目标地址** | 直接连接MMSC URL | HTTP请求发送到代理服务器，代理转发到MMSC |
| **DNS解析** | 客户端解析MMSC域名 | 代理服务器负责解析（或客户端解析代理域名） |
| **典型运营商** | 大部分国际运营商 | 部分CDMA运营商（如Verizon） |

```java
// MmsHttpClient.execute()中的代理选择逻辑
Proxy proxy = Proxy.NO_PROXY;
if (isProxySet) {
    // 代理模式下，通过MMS网络解析代理地址
    proxy = new Proxy(Proxy.Type.HTTP,
            new InetSocketAddress(mNetwork.getByName(proxyHost), proxyPort));
}
final URL url = new URL(urlString);
connection = (HttpURLConnection) mNetwork.openConnection(url, proxy);
```

关键点：代理地址通过`mNetwork.getByName(proxyHost)`解析而非全局DNS，确保在MMS专用数据链路上解析。

### 5.2 HTTP POST请求构造

MMS发送使用HTTP POST方法，请求体的格式遵循OMA MMS规范。`MmsHttpClient.execute()`方法完成了整个HTTP请求的构造。

**请求Headers详解：**

| Header名称 | 常量 | 值/来源 | 说明 |
|-----------|------|--------|------|
| `Accept` | `HEADER_ACCEPT` | `*/*, application/vnd.wap.mms-message, application/vnd.wap.sic` | 接受MMS PDU和SI（Service Indication）内容类型 |
| `Accept-Language` | `HEADER_ACCEPT_LANGUAGE` | 动态生成（如`zh-CN, en-US`） | 当前Locale + US Locale组合，适配运营商网关的语言偏好 |
| `User-Agent` | `HEADER_USER_AGENT` | `mmsConfig.getString(MMS_CONFIG_USER_AGENT)` | 运营商定义的UA字符串，TelephonyManager.getMmsUserAgent()提供默认值 |
| `Content-Type` | `HEADER_CONTENT_TYPE` | `application/vnd.wap.mms-message` 或 `application/vnd.wap.mms-message; charset=utf-8` | 取决于`MMS_CONFIG_SUPPORT_HTTP_CHARSET_HEADER`配置 |
| `x-wap-profile` | 动态 | `mmsConfig.getString(MMS_CONFIG_UA_PROF_URL)` | UA Profile URL，标识设备能力。Header名由`MMS_CONFIG_UA_PROF_TAG_NAME`配置，默认`x-wap-profile` |
| `Connection` | `HEADER_CONNECTION` | `close` | 可选，由`KEY_MMS_CLOSE_CONNECTION_BOOL`控制，禁用HTTP Keep-Alive |
| Extra Headers | 动态 | `mmsConfig.getString(MMS_CONFIG_HTTP_PARAMS)` | 运营商自定义Header，格式`key1:value1\|key2:value2` |

**Extra Headers与宏替换：**

`addExtraHeaders`方法解析运营商自定义的HTTP参数，支持宏替换：

| 宏名称 | 宏语法 | 替换为 | 用途 |
|--------|--------|--------|------|
| `LINE1` | `##LINE1##` | TelephonyManager.getLine1Number() | 设备电话号码（含国家码） |
| `LINE1NOCOUNTRYCODE` | `##LINE1NOCOUNTRYCODE##` | 号码去国家码部分 | 设备号码（不含国家码） |
| `NAI` | `##NAI##` | TelephonyManager.getNai() + NAI_SUFFIX（Base64编码） | Network Access Identifier，Sprint运营商用于认证 |

宏解析使用正则表达式`##(\\S+)##`匹配，通过`resolveMacro`方法替换为实际值。

**请求Body写入：**

```java
// POST方法下的PDU写入
if (METHOD_POST.equals(method)) {
    if (pdu == null || pdu.length < 1) {
        throw new MmsHttpException(0, "Sending empty PDU");
    }
    connection.setDoOutput(true);
    connection.setRequestMethod(METHOD_POST);
    // Content-Type: 区分是否带charset
    if (mmsConfig.getBoolean(SmsManager.MMS_CONFIG_SUPPORT_HTTP_CHARSET_HEADER)) {
        connection.setRequestProperty(HEADER_CONTENT_TYPE,
                HEADER_VALUE_CONTENT_TYPE_WITH_CHARSET);  // 带 charset=utf-8
    } else {
        connection.setRequestProperty(HEADER_CONTENT_TYPE,
                HEADER_VALUE_CONTENT_TYPE_WITHOUT_CHARSET); // 不带 charset
    }
    connection.setFixedLengthStreamingMode(pdu.length);
    // 写入PDU二进制数据
    final OutputStream out = new BufferedOutputStream(connection.getOutputStream());
    out.write(pdu);
    out.flush();
    out.close();
}
```

关键设计决策：
- 使用`setFixedLengthStreamingMode(pdu.length)`而非`setChunkedStreamingMode`，因为许多运营商的MMS网关不支持分块传输编码
- 使用`BufferedOutputStream`包装输出流以提高写入性能
- PDU数据以原始二进制字节流写入，不做Base64或URL编码

### 5.3 IPv4等待机制

在IPv6-only网络环境下，如果MMSC URL使用IPv4地址字面量（如`http://10.0.0.1:10021/mmsc`），网络可能尚未为该连接分配IPv4地址（NAT64尚未就绪）。`maybeWaitForIpv4`方法提供了等待机制。

```java
private void maybeWaitForIpv4(final String requestId, final URL url) {
    // 1. 检查URL的Host部分是否为IPv4字面量
    Inet4Address ipv4Literal = null;
    try {
        ipv4Literal = (Inet4Address) InetAddress.parseNumericAddress(url.getHost());
    } catch (IllegalArgumentException | ClassCastException e) {
        // 不是IP地址（是域名），跳过
    }
    if (ipv4Literal == null) {
        return;  // 不是IPv4地址，无需等待
    }

    // 2. 最多尝试15次，每次间隔1秒
    for (int i = 0; i < IPV4_WAIT_ATTEMPTS; i++) {
        final LinkProperties lp = mConnectivityManager.getLinkProperties(mNetwork);
        if (lp != null) {
            if (!lp.isReachable(ipv4Literal)) {
                LogUtil.w(requestId, "HTTP: IPv4 not yet provisioned");
                Thread.sleep(IPV4_WAIT_DELAY_MS);  // 1000ms
            } else {
                LogUtil.i(requestId, "HTTP: IPv4 provisioned");
                break;  // IPv4可达，退出等待
            }
        } else {
            LogUtil.w(requestId, "HTTP: network disconnected, skip ipv4 check");
            break;  // 网络已断开，无需等待
        }
    }
}
```

**参数配置：**

| 参数 | 常量 | 值 | 说明 |
|------|------|---|------|
| 最大等待次数 | `IPV4_WAIT_ATTEMPTS` | 15 | 最多检测15次 |
| 每次间隔 | `IPV4_WAIT_DELAY_MS` | 1000ms (1s) | 每次检测间隔1秒 |
| 最大总等待时间 | - | 15秒 | 15次 x 1秒 |

**等待策略分析：**

此机制通过`LinkProperties.isReachable(Inet4Address)`判断IPv4目标是否可达。该检查基于LinkProperties中的路由信息（如默认网关、DNS服务器等）间接判断IPv4连通性，而非实际发送ICMP探测包。在IPv6-only网络中，`isReachable`在CLAT（464XLAT）或NAT64配置就绪后会返回true。

如果15秒后IPv4仍未就绪，方法正常返回，后续的HTTP连接将尝试执行并大概率失败（触发MmsHttpException重试机制）。

## 第6章 MMS请求队列管理与错误处理

### 6.1 MmsService请求队列机制

`MmsService`采用双队列架构管理MMS请求：一个**运行队列**和**一个待执行队列**（pending queue），配合SIM互斥约束确保同一时刻只有一个SIM的MMS请求在执行。

**双队列架构：**

```
                    MmsService
    ┌──────────────────────────────────────────────┐
    │                                              │
    │  mPendingSimRequestQueue (ArrayDeque)        │  ← 等待SIM可用的请求
    │  ┌─────┐ ┌─────┐ ┌─────┐                    │
    │  │Req3 │ │Req4 │ │Req5 │                    │
    │  └─────┘ └─────┘ └─────┘                    │
    │                                              │
    │  mRunningRequestExecutors[0]  ← Send线程池    │  ← QUEUE_INDEX_SEND
    │  (FixedThreadPool, size=4)                  │
    │  mRunningRequestExecutors[1]  ← Download线程池│  ← QUEUE_INDEX_DOWNLOAD
    │  (FixedThreadPool, size=4)                  │
    │                                              │
    │  mCurrentSubId: 当前执行的SIM ID              │
    │  mRunningRequestCount: 当前运行中的请求数     │
    └──────────────────────────────────────────────┘
```

**队列类型常量：**

```java
public static final int QUEUE_INDEX_SEND = 0;      // 发送队列
public static final int QUEUE_INDEX_DOWNLOAD = 1;  // 下载队列
public static final int THREAD_POOL_SIZE = 4;      // 每个队列的线程池大小
```

**addSimRequest的核心调度逻辑：**

```java
// MmsService.addSimRequest()
public void addSimRequest(MmsRequest request) {
    synchronized (this) {
        if (mPendingSimRequestQueue.size() > 0 ||
                (mRunningRequestCount > 0 && request.getSubId() != mCurrentSubId)) {
            // 条件1: pending队列非空 → 新请求必须排队
            // 条件2: 有请求在运行 且 新请求的SIM不同 → 必须等待当前SIM完成
            mPendingSimRequestQueue.add(request);
        } else {
            // 可立即执行：pending队列为空 且 (无请求在运行 或 SIM相同)
            addToRunningRequestQueueSynchronized(request);
        }
    }
}
```

**SIM互斥约束的时序分析：**

```
时间 →
t0: SIM1-Req1 入队 → 运行 (mCurrentSubId=SIM1, count=1)
t1: SIM2-Req2 入队 → pending (SIM2 != SIM1, count=1)  ← 被阻塞
t2: SIM1-Req3 入队 → pending (pending非空)              ← 即使SIM相同也被阻塞（保序）
t3: SIM1-Req1 完成 → count=0 → movePendingSimRequests
t4: SIM2-Req2 出队运行 (mCurrentSubId=SIM2, count=1)
t5: SIM1-Req3 仍在pending (SIM1 != SIM2)               ← 继续被阻塞
t6: SIM2-Req2 完成 → count=0 → movePendingSimRequests
t7: SIM1-Req3 出队运行 (mCurrentSubId=SIM1, count=1)
```

注意：pending队列中的请求**不会被重排序**。即使后续到达的请求SIM与当前运行SIM相同，它也必须排在pending队列中不同SIM请求之后。源码注释明确指出此设计选择。

**movePendingSimRequestsToRunningSynchronized的批量处理：**

```java
private void movePendingSimRequestsToRunningSynchronized() {
    mCurrentSubId = SubscriptionManager.INVALID_SUBSCRIPTION_ID;  // 重置
    while (mPendingSimRequestQueue.size() > 0) {
        final MmsRequest request = mPendingSimRequestQueue.peek();
        if (!SubscriptionManager.isValidSubscriptionId(mCurrentSubId)
                || mCurrentSubId == request.getSubId()) {
            // 首次或同SIM → 出队执行
            mPendingSimRequestQueue.remove();
            addToRunningRequestQueueSynchronized(request);
        } else {
            // 遇到不同SIM → 停止，保留在队列中
            break;
        }
    }
}
```

此方法会**批量地将同一SIM的连续请求全部出队执行**，直到遇到不同SIM的请求为止。这最大化了同SIM请求的并发执行效率。

### 6.2 重试机制详解

MMS HTTP请求的重试机制在`MmsRequest.execute()`中实现，采用**指数退避**策略。

**重试参数：**

| 参数 | 值 | 说明 |
|------|---|------|
| `RETRY_TIMES` | 3 | 最大重试次数（含首次尝试，即最多执行3次） |
| 初始退避时间 | 2秒 | 第一次重试前的等待时间 |
| 退避策略 | `retryDelaySecs <<= 1`（左移1位，即x2） | 指数退避：2s → 4s → 8s |
| 最大退避时间 | 8秒 | 第三次重试前的等待时间 |

**重试流程源码分析：**

```java
// MmsRequest.execute()中的重试循环
long retryDelaySecs = 2;     // 初始退避2秒
for (retryId = 0; retryId < RETRY_TIMES; retryId++) {
    httpStatusCode = 0;
    try {
        // 1. 注册数据连接状态监听
        listenToDataConnectionState(connectionStateCallback);
        // 2. 获取网络
        currentState = MmsRequestState.AcquiringNetwork;
        int networkId = networkManager.acquireNetwork(requestId);
        // 3. 加载APN
        currentState = MmsRequestState.LoadingApn;
        ApnSettings apn = loadApn(...);
        // 4. 执行HTTP
        currentState = MmsRequestState.DoingHttp;
        response = doHttp(context, networkManager, apn);
        result = Activity.RESULT_OK;
        break;  // 成功 → 退出循环
    } catch (ApnException e) {
        result = SmsManager.MMS_ERROR_INVALID_APN;
        break;  // APN错误 → 不可重试，立即退出
    } catch (MmsNetworkException e) {
        result = SmsManager.MMS_ERROR_UNABLE_CONNECT_MMS;
        break;  // 网络获取失败 → 不可重试，立即退出
    } catch (MmsHttpException e) {
        result = SmsManager.MMS_ERROR_HTTP_FAILURE;
        httpStatusCode = e.getStatusCode();
        // 不break → 继续重试
    } catch (Exception e) {
        result = SmsManager.MMS_ERROR_UNSPECIFIED;
        break;  // 未知异常 → 不可重试，立即退出
    } finally {
        // 每次尝试后都释放网络（Download成功时延迟释放除外）
        networkManager.releaseNetwork(requestId,
                this instanceof DownloadRequest && result == Activity.RESULT_OK);
    }

    // 指数退避等待
    if (result != Activity.RESULT_CANCELED) {
        try {
            new CountDownLatch(1).await(retryDelaySecs, TimeUnit.SECONDS);
        } catch (InterruptedException e) { }
        retryDelaySecs <<= 1;  // 2 → 4 → 8
    }
}
```

**可重试 vs 不可重试错误分类：**

| 异常类型 | 是否重试 | 对应错误码 | 原因 |
|---------|---------|-----------|------|
| `MmsHttpException` | **重试** | `MMS_ERROR_HTTP_FAILURE` | HTTP层面的临时性错误（超时、5xx、网络波动），重试可能成功 |
| `ApnException` | **不重试** | `MMS_ERROR_INVALID_APN` | APN配置是静态的，重试不会改变结果 |
| `MmsNetworkException` | **不重试** | `MMS_ERROR_UNABLE_CONNECT_MMS` | 网络获取超时或SIM状态问题，重试无意义 |
| `Exception`（通用） | **不重试** | `MMS_ERROR_UNSPECIFIED` | 未知异常，风险不可控 |
| 卫星PDU过大 | **不重试** | `MMS_ERROR_TOO_LARGE_FOR_TRANSPORT` | 通过`break`立即退出循环 |

**重试时间线示例（全部3次均因HTTP失败重试）：**

```
t0:       第1次尝试 → HTTP失败
t0-t2s:   CountDownLatch.await(2s) — 退避等待
t2s:      第2次尝试 → HTTP失败
t2s-t6s:  CountDownLatch.await(4s) — 退避等待
t6s:      第3次尝试 → HTTP失败
          循环结束，result = MMS_ERROR_HTTP_FAILURE
```

### 6.3 错误码体系

`SmsManager`中定义了完整的MMS错误码常量，覆盖了MMS发送/下载全流程中所有可能的失败场景。

**MMS_ERROR_* 完整列表：**

| 错误码常量 | 值 | 含义 | 触发场景 |
|-----------|---|------|---------|
| `MMS_ERROR_UNSPECIFIED` | 1 | 未指定错误 | 未知异常、通用失败兜底 |
| `MMS_ERROR_INVALID_APN` | 2 | APN无效 | `ApnException`，无法从数据库加载匹配的MMS APN |
| `MMS_ERROR_UNABLE_CONNECT_MMS` | 3 | 无法连接MMS网络 | `MmsNetworkException`，`MmsNetworkManager.acquireNetwork`超时 |
| `MMS_ERROR_HTTP_FAILURE` | 4 | HTTP请求失败 | `MmsHttpException`，HTTP非2xx响应或IO异常 |
| `MMS_ERROR_IO_ERROR` | 5 | IO错误 | PDU读取失败、`transferResponse`失败 |
| `MMS_ERROR_RETRY` | 6 | 重试错误 | 载波应用返回需要重试的状态码 |
| `MMS_ERROR_CONFIGURATION_ERROR` | 7 | 配置错误 | MMS配置未加载、`MmsConfigManager`返回null |
| `MMS_ERROR_NO_DATA_NETWORK` | 8 | 无数据网络 | MMS数据连接不可用（旧版错误码，已被`DATA_DISABLED`细分） |
| `MMS_ERROR_INVALID_SUBSCRIPTION_ID` | 9 | Subscription ID无效 | `!SubscriptionManager.isValidSubscriptionId(subId)` |

| `MMS_ERROR_MMS_DISABLED_BY_CARRIER` | 12 | 运营商禁用MMS | 载波配置`MMS_CONFIG_MMS_ENABLED`为false（需Flag `mmsDisabledError`） |

**错误码的产生位置映射：**

| 阶段 | 方法 | 可产生的错误码 |
|------|------|--------------|
| 请求预校验 | `sendMessageWithoutUpgrade` | `INVALID_SUBSCRIPTION_ID`, `INACTIVE_SUBSCRIPTION`, `CONFIGURATION_ERROR`, `MMS_DISABLED_BY_CARRIER`, `DATA_DISABLED` |
| PDU准备 | `prepareForHttpRequest` | `IO_ERROR`（PDU读取失败） |
| 网络获取 | `acquireNetwork` | `UNABLE_CONNECT_MMS` |
| APN加载 | `ApnSettings.load` | `INVALID_APN` |
| HTTP传输 | `doHttp` → `execute` | `HTTP_FAILURE`, `IO_ERROR` |
| 卫星检查 | `canTransferPayloadOnCurrentNetwork` | `TOO_LARGE_FOR_TRANSPORT` |
| 结果处理 | `processResult` | `IO_ERROR`（transferResponse失败） |

**载波应用错误码转换：**

`MmsRequest`中提供了`toSmsManagerResultForOutboundMms`和`toSmsManagerResultForInboundMms`两个静态方法，将`CarrierMessagingService`定义的状态码映射到平台错误码。这使得载波应用的错误报告与平台错误码体系统一。

### 6.4 异常报告与监控

Android MMS框架集成了三层监控体系：AnomalyReporter、MonitorTelephonyCallback和Statsd Atom。

**1. AnomalyReporter（异常上报）：**

AnomalyReporter是Android Telephony框架提供的异常检测与上报机制。在MMS中有两个上报点：

```java
// 上报点1: 快速失败场景（预校验阶段）
// MmsService.mStub.handleError()
private void handleError(PendingIntent pendingIntent, int resultCode, MmsStats mmsStats) {
    sendErrorInPendingIntent(pendingIntent, resultCode);
    mmsStats.addAtomToStorage(resultCode);
    String message = "MMS failed with error " + resultCode;
    AnomalyReporter.reportAnomaly(MmsConstants.MMS_ANOMALY_UUID, message);
}

// 上报点2: 执行阶段（HTTP失败等）
// MmsRequest.reportPossibleAnomaly()
private void reportPossibleAnomaly(int result, int httpStatusCode) {
    switch (result) {
        case SmsManager.MMS_ERROR_HTTP_FAILURE:
            if (isPoorSignal()) break;  // 弱信号不报异常
        case SmsManager.MMS_ERROR_INVALID_APN:
        case SmsManager.MMS_ERROR_UNABLE_CONNECT_MMS:
        case SmsManager.MMS_ERROR_UNSPECIFIED:
        case SmsManager.MMS_ERROR_IO_ERROR:
            String message = "MMS failed";
            reportAnomaly(message, generateUUID(result, httpStatusCode));
            break;
    }
}
```

异常上报的弱信号排除逻辑：当错误类型为`MMS_ERROR_HTTP_FAILURE`时，如果当前处于弱信号环境（WiFi RSSI级别 <= 2 或蜂窝信号级别 <= 2），则不上报异常，因为弱信号下的失败是预期行为。

UUID生成策略：

```java
private UUID generateUUID(int result, int httpStatusCode) {
    return new UUID(
        MmsConstants.MMS_ANOMALY_UUID.getMostSignificantBits(),     // 固定高64位
        MmsConstants.MMS_ANOMALY_UUID.getLeastSignificantBits()      // 基础低64位
            + ((long)httpStatusCode << 32) + (long)result           // 编码错误信息
    );
}
```

基础UUID为`e4330975-17be-43b7-87d6-d9f281d33278`，通过在低64位编码`httpStatusCode`和`result`来区分不同的失败类型。

**2. MonitorTelephonyCallback（数据连接状态监控）：**

`MmsRequest.MonitorTelephonyCallback`是每个请求执行期间注册的临时回调，用于跟踪数据连接状态变化：

```java
class MonitorTelephonyCallback extends TelephonyCallback implements
        TelephonyCallback.PreciseDataConnectionStateListener {
    private final SparseArray<ApnSetting> mNetworkIdToApn = new SparseArray<>(2);

    @Override
    public void onPreciseDataConnectionStateChanged(
            PreciseDataConnectionState connectionState) {
        ApnSetting apnSetting = connectionState.getApnSetting();
        if (apnSetting != null && (apnSetting.getApnTypeBitmask() & ApnSetting.TYPE_MMS) != 0) {
            mLastConnectionFailure = connectionState.getLastCauseCode();
            synchronized (mLock) {
                mNetworkIdToApn.put(connectionState.getNetId(), apnSetting);
            }
        }

```

此回调的两个核心功能：
1. **APN信息获取**：从`PreciseDataConnectionState`中提取网络ID到APN设置的映射，用于替代或补充数据库APN加载

## 第7章 数据链路释放与收尾流程

### 7.1 网络释放触发机制

MMS请求执行完毕后，网络释放在`MmsRequest.execute()`的`finally`块中触发，确保无论请求成功或失败都会执行释放：

```java
// MmsRequest.execute()中的finally块
finally {
    // 释放MMS网络
    // 参数shouldDelayRelease: DownloadRequest成功时为true，其他情况为false
    networkManager.releaseNetwork(requestId,
            this instanceof DownloadRequest
                    && result == Activity.RESULT_OK);
    // 停止数据连接状态监听
    stopListeningToDataConnectionState(connectionStateCallback);
}
```

释放时机的关键设计：

| 场景 | shouldDelayRelease | 原因 |
|------|------------------|------|
| SendRequest成功 | false | 发送完成后无需保留网络，立即释放 |
| SendRequest失败 | false | 失败后无需保留网络 |
| DownloadRequest成功 | **true** | 下载成功后可能需要发送AcknowledgeInd（使用同一网络） |
| DownloadRequest失败 | false | 下载失败，无需发送ACK |

DownloadRequest延迟释放的设计考虑：MMS下载完成后通常需要发送`AcknowledgeInd`通知MMSC已成功接收，此ACK需要使用同一个MMS数据连接。延迟释放确保网络在ACK发送前不被回收。

**注意：** 在重试循环中，每次尝试（包括成功的尝试，因为成功后会break退出循环）都会执行finally块。但只有DownloadRequest成功时才延迟释放，因此SendRequest成功时网络在退出循环前就被释放。

### 7.2 MmsNetworkManager释放流程

`MmsNetworkManager`使用**引用计数**机制管理网络生命周期，支持延迟释放。

**引用计数模型：**

```java
// acquireNetwork()中递增
mMmsRequestCount += 1;
// 移除延迟释放任务（因为有了新的请求）
mReleaseHandler.removeCallbacks(mNetworkReleaseTask);

// releaseNetwork()中递减
mMmsRequestCount -= 1;
if (mMmsRequestCount < 1) {
    if (shouldDelayRelease) {
        // 延迟释放：postDelayed到主线程Handler
        mReleaseHandler.removeCallbacks(mNetworkReleaseTask);
        mReleaseHandler.postDelayed(mNetworkReleaseTask,
                mNetworkReleaseTimeoutMillis);
    } else {
        // 立即释放
        releaseRequestLocked(mNetworkCallback);
    }
}
```

**引用计数的并发场景分析：**

```
场景1: 多个请求并发执行
  Req1.acquire → count=1
  Req2.acquire → count=2
  Req1.release(shouldDelay=false) → count=1, 不释放
  Req2.release(shouldDelay=false) → count=0, 立即释放

场景2: DownloadRequest成功后立即有新请求
  DownloadReq.acquire → count=1
  DownloadReq.release(shouldDelay=true) → count=0, postDelayed(releaseTask, timeout)
  NewReq.acquire → count=1, removeCallbacks(releaseTask) ← 取消了延迟释放
  NewReq.release(shouldDelay=false) → count=0, 立即释放

场景3: DownloadRequest成功后无新请求
  DownloadReq.acquire → count=1
  DownloadReq.release(shouldDelay=true) → count=0, postDelayed(releaseTask, timeout)
  ... timeout时间到 → releaseTask.run() → count<1 → 释放
```

**延迟释放的超时配置：**

延迟释放的超时时间由运营商配置决定：

```java
// 从CarrierConfigManager读取
mNetworkReleaseTimeoutMillis =
    config.getInt(CarrierConfigManager.KEY_MMS_NETWORK_RELEASE_TIMEOUT_MILLIS_INT);
```

该值通过`ACTION_CARRIER_CONFIG_CHANGED`广播动态更新，`mCarrierConfigChangedReceiver`负责监听并重新加载。

**releaseRequestLocked的核心操作：**

```java
private void releaseRequestLocked(ConnectivityManager.NetworkCallback callback) {
    if (callback != null) {
        final ConnectivityManager connectivityManager = getConnectivityManager();
        try {
            connectivityManager.unregisterNetworkCallback(callback);
        } catch (IllegalArgumentException e) {
            // 捕获异常：requestNetwork可能因RemoteException静默失败
            // callback无效时unregister会抛IllegalArgumentException
        }
    }
    resetLocked();  // 重置所有内部状态
}

private void resetLocked() {
    mNetworkCallback = null;
    mNetwork = null;
    mMmsRequestCount = 0;
    mMmsHttpClient = null;
}
```

**网络请求超时配置：**

| 参数 | 常量/默认值 | 说明 |
|------|-----------|------|
| 网络请求超时 | `DEFAULT_MMS_SERVICE_NETWORK_REQUEST_TIMEOUT_MILLIS` = 30分钟 | `ConnectivityManager.requestNetwork`的超时参数 |
| 额外等待超时 | `ADDITIONAL_NETWORK_ACQUIRE_TIMEOUT_MILLIS` = 5秒 | 在网络请求超时后的额外等待缓冲 |
| 可通过DeviceConfig覆盖 | `mms_service_network_request_timeout_millis` | 通过`DeviceConfig.setProperty`动态配置 |

### 7.3 SIM卡状态变更处理

`MmsNetworkManager`在发起网络请求后会注册`ACTION_SIM_CARD_STATE_CHANGED`广播接收器，用于处理SIM卡热拔插场景：

```java
private final BroadcastReceiver mSimCardStateChangedReceiver =
        new BroadcastReceiver() {
    @Override
    public void onReceive(Context context, Intent intent) {
        final int simState = intent.getIntExtra(
                TelephonyManager.EXTRA_SIM_STATE,
                TelephonyManager.SIM_STATE_UNKNOWN);
        final int phoneId = intent.getIntExtra(
                PhoneConstants.PHONE_KEY,
                SubscriptionManager.INVALID_PHONE_INDEX);

        if (mPhoneId == phoneId && simState == TelephonyManager.SIM_STATE_ABSENT) {
            // 当前SIM被拔出 → 立即释放网络请求并唤醒等待线程
            synchronized (MmsNetworkManager.this) {
                releaseRequestLocked(mNetworkCallback);
                MmsNetworkManager.this.notifyAll();
            }
        }
    }
};
```

**生命周期管理：**

- **注册时机**：`acquireNetwork()`中首次发起网络请求时注册
- **注销时机**：`acquireNetwork()`的`wait`返回后注销（无论成功或超时）
- **作用域**：仅在`acquireNetwork()`的等待期间活跃

**SIM卡状态码：**

| 状态码 | 常量 | 说明 |
|--------|------|------|
| `SIM_STATE_UNKNOWN` | -1 | 状态未知 |
| `SIM_STATE_ABSENT` | 1 | SIM卡不存在 |
| `SIM_STATE_CARD_IO_ERROR` | 2 | SIM卡IO错误 |
| `SIM_STATE_CARD_RESTRICTED` | 3 | SIM卡受限 |
| `SIM_STATE_PRESENT` | 5 | SIM卡在位 |

只有`SIM_STATE_ABSENT`触发网络释放，其他状态变化不产生任何操作。

**载波配置变更监听：**

`mCarrierConfigChangedReceiver`监听`ACTION_CARRIER_CONFIG_CHANGED`，当订阅ID匹配时触发`handleCarrierConfigChanged()`重新加载`mNetworkReleaseTimeoutMillis`。此接收器在构造函数中注册，生命周期与`MmsNetworkManager`一致。

### 7.4 完整流程收尾

请求执行完成后的收尾工作在`MmsRequest.processResult()`中完成，包含4个关键步骤。

**processResult完整流程：**

```java
public void processResult(Context context, int result, byte[] response,
        int httpStatusCode, boolean handledByCarrierApp, int retryId) {
    // 步骤1: 持久化消息到数据库
    final Uri messageUri = persistIfRequired(context, result, response);

    // 步骤2: 更新请求状态
    currentState = result == Activity.RESULT_OK
            ? MmsRequestState.Success
            : MmsRequestState.Failure;

    // 步骤3: 通过PendingIntent通知调用方
    final PendingIntent pendingIntent = getPendingIntent();
    if (pendingIntent != null) {
        Intent fillIn = new Intent();
        // 3a. 传递响应PDU
        if (response != null) {
            succeeded = transferResponse(fillIn, response);
        }
        // 3b. 传递消息Uri
        if (messageUri != null) {
            fillIn.putExtra("uri", messageUri.toString());
        }
        // 3c. 传递HTTP状态码（仅HTTP失败时）
        if (result == SmsManager.MMS_ERROR_HTTP_FAILURE && httpStatusCode != 0) {
            fillIn.putExtra(SmsManager.EXTRA_MMS_HTTP_STATUS, httpStatusCode);
        }
        // 3d. 传递连接失败原因码
        fillIn.putExtra(EXTRA_LAST_CONNECTION_FAILURE_CAUSE_CODE, mLastConnectionFailure);
        // 3e. 传递是否由载波应用处理
        fillIn.putExtra(EXTRA_HANDLED_BY_CARRIER_APP, handledByCarrierApp);

        // 3f. 上报异常
        reportPossibleAnomaly(result, httpStatusCode);
        // 3g. 发送PendingIntent
        pendingIntent.send(context, result, fillIn);
        // 3h. 上报Statsd Atom
        mMmsStats.addAtomToStorage(result, retryId, handledByCarrierApp,
                mMessageId, getPduLength(result, response), httpStatusCode);
    }

    // 步骤4: 撤销URI权限
    revokeUriPermission(context);
}
```

**PendingIntent返回的Extras汇总：**

| Extra Key | 类型 | 条件 | 说明 |
|-----------|------|------|------|
| `android.telephony.extra.MMS_DATA` | `byte[]` | response != null | SendConf PDU原始字节（仅发送） |
| `uri` | `String` | messageUri != null | 持久化后的消息Uri（如`content://mms/sent/123`） |
| `android.telephony.extra.MMS_HTTP_STATUS` | `int` | HTTP失败且状态码非0 | MMSC返回的HTTP状态码（如500、503） |
| `android.telephony.extra.LAST_CONNECTION_FAILURE_CAUSE_CODE` | `int` | 始终 | 数据连接最后的失败原因码（来自MonitorTelephonyCallback） |
| `android.telephony.extra.HANDLED_BY_CARRIER_APP` | `boolean` | 始终 | 是否由载波应用完成发送/下载 |

**URI权限撤销（SendRequest实现）：**

```java
@Override
protected void revokeUriPermission(Context context) {
    if (mPduUri != null) {
        context.revokeUriPermission(mPduUri, Intent.FLAG_GRANT_READ_URI_PERMISSION);
    }
}
```

发送方应用在创建PDU Uri时通过`FLAG_GRANT_READ_URI_PERMISSION`授权MmsService读取PDU数据。请求完成后（无论成功或失败）必须撤销此权限，否则会留下安全漏洞——MmsService可能在后续请求中仍然能读取该Uri指向的数据。

**processResult中的transferResponse：**

对于SendRequest，`transferResponse`将SendConf PDU字节直接附加到Intent extras中：

```java
@Override
protected boolean transferResponse(Intent fillIn, byte[] response) {
    if (response != null) {
        fillIn.putExtra(SmsManager.EXTRA_MMS_DATA, response);
    }
    return true;
}
```

如果transferResponse返回false（某些实现中可能发生），processResult会将result改写为`MMS_ERROR_IO_ERROR`：

```java
if (!succeeded) {
    result = SmsManager.MMS_ERROR_IO_ERROR;
}
```

### 8.2 关键Log TAG与adb logcat过滤命令

**核心Log TAG：**

| TAG | 类 | 说明 |
|-----|---|------|
| `MmsService` | `LogUtil` | 所有MMS服务的日志，包括请求调度、网络管理、HTTP传输 |

`LogUtil`统一使用TAG `"MmsService"`，通过`Rlog`（Telephony专用Log类）输出。日志格式为`[requestId] message`，其中requestId包含类名、hashCode、messageId、subId和当前状态：

```
SendRequest@1a2b3c4 {x-message-id:42} subId: 1 currentState: "DoingHttp" result: 0
```

**按过滤场景的adb命令：**

```bash
adb logcat -s MmsService:V

adb logcat -s MmsService:W

adb logcat -s MmsService:V | grep "x-message-id:42"

adb logcat -s MmsService:V | grep "HTTP:"

adb logcat -s MmsService:V | grep -E "(acquire|release|Network|APN)"

adb logcat -s MmsService:V | grep -E "(pending|running|queue|subId=)"

adb logcat -s MmsService:V | grep -E "(retry|Retry|HTTP failure)"

adb logcat -s MmsService:V | grep "IPv4"

adb logcat -s MmsService:V | grep -i "anomaly"

adb logcat -s MmsService:V | grep -i "satellite"
```

**关键日志节点与对应输出示例：**

| 阶段 | 日志前缀 | 示例输出 |
|------|---------|---------|
| 请求入队 | `Current running=..., current subId=..., pending=...` | `Current running=1, current subId=1, pending=0` |
| PDU读取 | `persistIfRequired` / `readPduFromContentUri` | `persistIfRequired. {x-message-id:42}` |
| 网络获取 | `MmsNetworkManager: start new network request` / `already available` | `MmsNetworkManager: already available` |
| 网络超时 | `MmsNetworkManager: timed out with networkRequestTimeoutMillis=...` | `timed out with networkRequestTimeoutMillis=1800000` |
| HTTP开始 | `HTTP: POST ...` | `HTTP: POST http://mmsc.carrier.com[30], PDU size=12345` |

**URL脱敏机制：**

非VERBOSE日志级别下，MMSC URL被脱敏为仅显示协议+主机+原始URL长度：

```java
// MmsHttpClient.redactUrlForNonVerbose()
// 输入: "http://mmsc.carrier.com:10021/mmsc/send"
```

只有在`VERBOSE`级别日志中才会输出完整URL。启用VERBOSE日志命令：


### 8.3 常见问题排查思路

**问题1: 网络获取失败（MMS_ERROR_UNABLE_CONNECT_MMS）**

**症状：** 日志中出现`MmsNetworkManager: timed out`或`Acquiring network failed`

**排查步骤：**

1. 检查SIM卡状态：
   ```bash
   adb shell dumpsys isub        # 查看Subscription状态
   adb shell dumpsys telephony.registry  # 查看Telephony注册状态
   ```

2. 检查MMS数据是否启用：
   ```bash
   adb shell settings get global mms_data_enabled
   ```

3. 检查APN配置：
   ```bash
   adb shell content query --uri content://telephony/carriers/current
   ```

4. 查看网络请求日志：
   ```bash
   adb logcat -s MmsService:V ConnectivityService:V | grep -E "(MMS|requestNetwork)"
   ```

5. 常见原因：
   - 飞行模式开启
   - 数据漫游关闭但在漫游网络
   - SIM卡未就绪（`SIM_STATE_ABSENT`）
   - `isDataEnabledForApn(ApnSetting.TYPE_MMS)`返回false

**问题2: HTTP超时或连接失败（MMS_ERROR_HTTP_FAILURE）**

**症状：** 日志中出现`HTTP: IO failure`或`HTTP: 500/503/504`

**排查步骤：**

1. 确认MMSC URL可达性：
   ```bash
   adb logcat -s MmsService:V | grep "HTTP:"
   ```
   检查POST请求的目标URL是否正确

2. 检查是否在弱信号区域：
   ```bash
   adb logcat -s MmsService:V | grep -i "signal\|poor"
   ```
   如果出现`Poor Signal`日志，说明信号弱导致HTTP失败

3. 检查IPv4等待：
   ```bash
   adb logcat -s MmsService:V | grep "IPv4"
   ```
   如果大量出现`IPv4 not yet provisioned`且最终超时，说明CLAT/NAT64配置有问题

4. 检查HTTP状态码：
   ```bash
   adb logcat -s MmsService:V | grep "HTTP: [0-9]"
   ```
   常见状态码含义：
   - `200 OK`：成功，但SendConf中ResponseStatus非128
   - `400 Bad Request`：PDU格式错误或Header不正确
   - `403 Forbidden`：运营商认证失败（检查NAI/##LINE1##宏）
   - `404 Not Found`：MMSC URL错误
   - `500 Internal Server Error`：MMSC内部错误，可能需要重试
   - `503 Service Unavailable`：MMSC过载

5. 检查Private DNS冲突：
   ```bash
   adb shell settings get global private_dns_mode
   ```
   如果设置为`hostname`（如dns.google），MMSC域名可能无法解析。`getPrivateDnsBypassingCopy()`应能绕过此问题

**问题3: APN配置错误（MMS_ERROR_INVALID_APN）**

**症状：** 日志中出现`APN failure`或`No match with APN name`

**排查步骤：**

1. 查看当前活跃APN：
   ```bash
   adb shell dumpsys telephony.registry | grep -A 20 "DataConnection"
   adb shell content query --uri content://telephony/carriers/preferapn
   ```

2. 检查APN加载过程：
   ```bash
   adb logcat -s MmsService:V | grep -E "(APN|apn|mmsc)"
   ```
   查看加载了哪个APN以及MMSC URL是什么

3. 常见原因：
   - 未配置MMS APN（运营商APN列表中缺少`mms`类型的APN）
   - MMSC URL为空或格式错误
   - APN名称与数据库不匹配（`ApnSettings.load`会先按名称匹配，失败后回退到无名称匹配）

4. 确认APN设置：
   ```bash
   adb shell content query --uri content://telephony/carriers --where "type LIKE '%mms%'"
   ```

**问题4: PDU过大（MMS_ERROR_TOO_LARGE_FOR_TRANSPORT）**

**症状：** 日志中出现`PDU too large for satellite`

**排查步骤：**

1. 检查是否通过卫星传输：
   ```bash
   adb logcat -s MmsService:V | grep -i "satellite"
   ```

2. 查看PDU大小和限制：
   ```bash
   adb logcat -s MmsService:V | grep "payloadSize\|maxPduSize"
   ```

3. 默认卫星PDU大小限制：3KB（`SATELLITE_MMS_SIZE_LIMIT = 3 * 1024`）。可通过`CarrierConfigManager.KEY_MMS_MAX_NTN_PAYLOAD_SIZE_BYTES_INT`配置

**问题5: MMS被禁用（MMS_ERROR_DATA_DISABLED / MMS_ERROR_MMS_DISABLED_BY_CARRIER）**

**排查步骤：**

1. 检查MMS开关：
   ```bash
   adb shell settings get global mms_data_enabled
   ```

2. 检查运营商配置：
   ```bash
   adb shell dumpsys carrier_config | grep -i "mms"
   ```

3. 查看广播设置提示：
   ```bash
   adb logcat -s MmsService:V | grep "ACTION_ENABLE_MMS_DATA_REQUEST"
   ```

**问题6: 消息持久化失败**

**排查步骤：**

```bash
adb logcat -s MmsService:V | grep -E "(persist|SqliteWrapper|PduPersister)"
```

常见原因：
- `SmsApplication.shouldWriteMessageForPackageAsUser`返回false（非默认SMS应用且auto-persisting关闭）
- PDU解析失败（`parse()`返回null）
- 数据库写入异常（`SQLiteException`）

**通用排查流程总结：**

```
1. adb logcat -s MmsService:V > mms_log.txt
2. 在日志中定位messageId → 追踪该请求的完整生命周期
3. 检查currentState流转：PrepareForHttpRequest → AcquiringNetwork → LoadingApn → DoingHttp → Success/Failure
4. 在失败状态前查找ERROR/WARN级别日志
5. 根据result错误码定位失败阶段
```
