---

## 1 USSD 概述

### 1.1 什么是 USSD

USSD（Unstructured Supplementary Service Data，非结构化补充服务数据）是一种基于 **GSM/UMTS/LTE 电路域信令通道**的实时交互式数据业务。它允许用户与运营商网络之间进行双向、会话式的文本数据交换。

与 SMS 的关键区别：

| 特性 | USSD | SMS |
|------|------|-----|
| 传输通道 | 信令通道（DCCH） | 业务通道（SDCCH/SACCH） |
| 连接方式 | 面向会话，实时交互 | 存储转发 |
| 响应速度 | 秒级实时响应 | 依赖网络拥塞情况 |
| 字符限制 | 最多 182 字符 | 单条 160 字符（7-bit） |
| 交互模式 | 支持多轮会话交互 | 单向发送 |
| 费用 | 通常免费或低费率 | 按条计费 |

### 1.2 常见应用场景

- **余额查询**：`*101#`、`*100#` 等
- **呼叫转移设置**：`**21*号码#`（开启）、`##21#`（取消）
- **套餐查询/办理**：运营商自定义代码
- **充值**：`*123*充值卡密码#`
- **隐藏工程模式**：`*#*#4636#*#*`（显示手机信息）

### 1.3 USSD 代码格式

标准 USSD 格式遵循以下模式：

```
*xxx*xx*xxx#       // 查询或操作代码
**xx*xxx#          // 激活服务
*#xx#              // 查询服务状态
##xxx#             // 取消服务
*#*#xxxx#*#*       // Secret Code（设备工程代码）
```

## 2 整体架构

Android USSD 流程贯穿 **应用层 → Framework 层 → RIL 层 → HAL/Modem**，其架构如下：

```
┌─────────────────────────────────────────────────────────────────────┐
│                          应用层 (Applications)                       │
│             Dialer / Phone App / SpecialCharSequenceMgr              │
└──────────────────────────┬──────────────────────────────────────────┘
                           │ Intent / Binder IPC
┌──────────────────────────▼──────────────────────────────────────────┐
│                       Framework 层 (Java)                            │
│       TelephonyManager → PhoneInterfaceManager → Phone/GsmCdma      │
│       Phone → GsmMmiCode → MmiCode → USSD 状态管理                   │
└──────────────────────────┬──────────────────────────────────────────┘
                           │ RIL_REQUEST_SEND_USSD / mUSSDRegistrant
┌──────────────────────────▼──────────────────────────────────────────┐
│                        RIL Java 层 (RIL.java)                        │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │       RadioVoiceProxy (AIDL/HIDL 双路径统一封装)               │  │
│  │   ┌─────────────────────────────────────────────────────┐     │  │
│  │   │  AIDL: android.hardware.radio.voice.IRadioVoice    │     │  │
│  │   │  HIDL: android.hardware.radio.V1_x.IRadioVoice     │     │  │
│  │   └─────────────────────────────────────────────────────┘     │  │
│  └───────────────────────────────────────────────────────────────┘  │
│       ↑ Solicited Response (sendUssd)                               │
│       ↓ Unsolicited Indication                                      │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │   VoiceIndication (AIDL)  /  RadioIndication (HIDL)           │  │
│  │   └── onUssd() → mRil.mUSSDRegistrant.notifyRegistrant()     │  │
│  └───────────────────────────────────────────────────────────────┘  │
└──────────────────────────┬──────────────────────────────────────────┘
                           │ AIDL binder / HIDL hwservicemanager
┌──────────────────────────▼──────────────────────────────────────────┐
│                        RIL Daemon (rild - C/C++)                     │
│           libril.so → Reference-RIL / Vendor RIL → AT/QMI           │
└──────────────────────────┬──────────────────────────────────────────┘
                           │ AT Commands / QMI / RIL Primitives
┌──────────────────────────▼──────────────────────────────────────────┐
│                          Modem / Baseband                            │
│                基带处理器 → 射频模块 → 运营商网络                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 3 详细流程分析

### 3.1 阶段一：用户输入与拨号应用处理

#### 3.1.1 输入捕获

用户在拨号盘输入以 `*` 或 `#` 开头的特殊代码（如 `*101#`），拨号应用（Dialer）的 `DialpadFragment` 捕获按键事件。

```java
// packages/apps/Dialer/src/.../DialpadFragment.java
@Override
public void onClick(View view) {
    // 按键输入传递给 DigitsEditText
    // 最终通过 PlaceCallIntentBuilder 构建拨号 Intent
}
```

#### 3.1.2 Secret Code 拦截

拨号应用首先通过 `SpecialCharSequenceMgr` 检查是否属于设备工程代码（以 `*#*#` 开头，`#*#*` 结尾）。

```java
// packages/apps/Phone/Common/.../SpecialCharSequenceMgr.java
static boolean handleSecretCode(Context context, String input) {
    // 匹配 *#*#<code>#*#* 格式
    if (input.matches("\\*#\\*#[0-9]+#\\*#\\*")) {
        Intent intent = new Intent(TelephonyIntents.SECRET_CODE_ACTION,
                Uri.parse("android_secret_code://" + input.substring(4, input.length() - 4)));
        context.sendBroadcast(intent);
        return true; // 已消费，不走 USSD 流程
    }
    return false;
}
```

> **注意**：`*#*#4636#*#*`（测试界面）、`*#06#`（IMEI 显示）等属于本地处理代码，**不经过运营商网络**。

#### 3.1.3 普通 USSD/MMI 代码传递

若输入不是 Secret Code，拨号应用将构建 `ACTION_CALL` 或 `ACTION_DIAL` Intent，通过 `OutgoingCallReceiver` 或 `TelecomManager` 传递给系统电话服务。

```java
Intent intent = new Intent(Intent.ACTION_CALL);
intent.setData(Uri.parse("tel:" + ussdCode));
startActivity(intent);
```

### 3.2 阶段二：Telecom 与 Telephony Framework 处理

#### 3.2.1 TelecomManager 路由

`TelecomManager` 接收到拨号请求后，由 `TelecomService` 决定如何处理该呼叫。对于 USSD 代码，系统识别其为 **MMI（Man-Machine Interface）代码**，而非普通语音呼叫。

```
TelecomManager.placeCall() 
    → TelecomService 
    → ConnectionService (TelephonyConnectionService)
    → TelephonyConnection
```

#### 3.2.2 Phone 接口处理

`TelephonyConnectionService` 获取对应的 `Phone` 实例（`GsmCdmaPhone` 或 `ImsPhone`），调用 `dial()` 方法。

```java
// frameworks/opt/telephony/.../GsmCdmaPhone.java
@Override
public Connection dial(String dialString, @NonNull DialArgs dialArgs) throws CallStateException {
    // 1. 检查是否是 MMI 代码
    if (handleInCallMmiCommands(dialString)) {
        return null;
    }
    
    // 2. 创建 GsmMmiCode 对象解析 MMI
    GsmMmiCode mmi = GsmMmiCode.newFromDialString(networkCapabilities, this, dialString,
            obtainMessage(EVENT_MMI_DONE), dialArgs.uusInfo);
    
    if (mmi != null) {
        // 3. 判断是 USSD 还是 Supplementary Service（SS）
        if (mmi.isUssdRequest() || mmi.isSsInfo()) {
            mmi.sendUssd(ussdCallback);  // 进入 USSD 流程
            return null;
        }
        // ... SS 处理（呼叫转移等）
    }
    
    // 4. 普通语音呼叫
    return mCT.dial(dialString, dialArgs);
}
```

### 3.3 阶段三：MMI 代码解析（GsmMmiCode）

`GsmMmiCode` 是解析和处理 MMI/USSD 代码的核心类。

#### 3.3.1 代码解析逻辑

```java
// frameworks/opt/telephony/.../GsmMmiCode.java
static GsmMmiCode newFromDialString(...) {
    GsmMmiCode ret = null;
    
    // 使用正则表达式匹配 MMI 模式
    Matcher m = sPatternSuppService.matcher(dialString);
    if (m.matches()) {
        ret = new GsmMmiCode(phone, handler);
        ret.mPoundString = makeEmptyNull(m.group(MATCH_GROUP_POUND_STRING));
        ret.mAction = makeEmptyNull(m.group(MATCH_GROUP_ACTION));
        ret.mSc = makeEmptyNull(m.group(MATCH_GROUP_SERVICE_CODE));
        ret.mSia = makeEmptyNull(m.group(MATCH_GROUP_SIA));
        ret.mSib = makeEmptyNull(m.group(MATCH_GROUP_SIB));
        ret.mSid = makeEmptyNull(m.group(MATCH_GROUP_SID));
        ret.mDialingNumber = makeEmptyNull(m.group(MATCH_GROUP_DIALING_NUMBER));
    }
    return ret;
}
```

#### 3.3.2 正则表达式模式

```java
// 匹配 *SC*SIA*SIB*SID# 或 *SC*SIA*SIB*SID#DN 格式
static Pattern sPatternSuppService = Pattern.compile(
    "((\\*|\\*\\*|\\*#|#)[0-9\\*#]+(#|\\*#)?)"
);
```

#### 3.3.3 区分 USSD 与 Supplementary Service

```java
boolean isUssdRequest() {
    // 以 * 或 # 开头，但没有明确的 SS Action（如 **、##、*#）
    return mSc != null && mAction == null;
}

boolean isSsInfo() {
    // 补充业务代码（呼叫转移、呼叫等待等）
    return mSc != null && (mAction != null || mDialingNumber != null);
}
```

### 3.4 阶段四：RIL Java 层请求发送

#### 3.4.1 GsmMmiCode 发起 USSD 请求

```java
// frameworks/opt/telephony/.../GsmMmiCode.java
void sendUssd(ResultReceiver receiver) {
    mState = State.PENDING;
    mReceiver = receiver;
    
    // 向 RIL 发送 USSD 请求
    mPhone.mCi.sendUSSD(mDialingNumber, 
        obtainMessage(EVENT_USSD_COMPLETE),
        obtainMessage(EVENT_USSD_CANCEL_COMPLETE));
}
```

#### 3.4.2 RIL.java 封装请求

```java
// frameworks/opt/telephony/.../RIL.java
@Override
public void sendUSSD(String ussd, Message result) {
    RadioVoiceProxy voiceProxy = getRadioServiceProxy(RadioVoiceProxy.class);
    if (!canMakeRequest("sendUSSD", voiceProxy, result, RADIO_HAL_VERSION_1_4)) {
        return;
    }

    RILRequest rr = obtainRequest(RIL_REQUEST_SEND_USSD, result,
            mRILDefaultWorkSource);

    boolean logUssd = TelephonyProperties.log_ussd_apdu().orElse(false);
    if (RILJ_LOGD) {
        riljLog(rr.serialString() + "> " + requestToString(rr.mRequest)
                + " ussd = " + logUssd);
    }

    radioServiceInvokeHelper(HAL_SERVICE_VOICE, rr, "sendUSSD", () -> {
        voiceProxy.sendUssd(rr.mSerial, RILUtils.convertNullToEmptyString(ussd));
    });
}
```

> **RIL_REQUEST_SEND_USSD** 的请求值为 `29`。与旧版直接操作 `IRadio` 不同，当前实现通过 `RadioVoiceProxy` 统一封装 AIDL/HIDL 双路径。

#### 3.4.3 RadioVoiceProxy 双路径封装

`RadioVoiceProxy` 负责屏蔽底层 HAL 类型差异，对外提供统一接口：

```java
// frameworks/opt/telephony/.../RadioVoiceProxy.java
public void sendUssd(int serial, String ussd) throws RemoteException {
    if (isEmpty()) return;
    if (isAidl()) {
        mVoiceProxy.sendUssd(serial, ussd);   // AIDL 路径
    } else {
        mRadioProxy.sendUssd(serial, ussd);   // HIDL 兼容路径
    }
}
```

- **AIDL 路径**：调用 `android.hardware.radio.voice.IRadioVoice.sendUssd()`
- **HIDL 路径**：调用 `android.hardware.radio.V1_x.IRadioVoice.sendUssd()`

### 3.5 阶段五：RIL Daemon 与 Modem 交互

#### 3.5.1 RIL Daemon 架构

```
┌──────────────────────────────────────────┐
│          RIL Java (system_server)        │
│              RIL.java                    │
└──────────────┬───────────────────────────┘
               │ HIDL / AIDL / Socket
┌──────────────▼───────────────────────────┐
│         RIL Daemon (rild - C/C++)        │
│  ┌────────────────────────────────────┐  │
│  │        libril.so / libreference     │  │
│  │  处理请求序列化、响应反序列化        │  │
│  └──────────────────┬─────────────────┘  │
│  ┌──────────────────▼─────────────────┐  │
│  │     Vendor RIL / Reference RIL      │  │
│  │   将 RIL 请求转换为 AT 命令或 QMI   │  │
│  └──────────────────┬─────────────────┘  │
└─────────────────────┼────────────────────┘
                      │ AT / QMI / RIL Primitives
┌─────────────────────▼────────────────────┐
│              Modem / Baseband            │
│     Qualcomm/Samsung/MTK 等基带芯片       │
└──────────────────────────────────────────┘
```

#### 3.5.2 AT 命令转换

对于大部分传统 Modem，RIL Daemon 将 `RIL_REQUEST_SEND_USSD` 转换为 **AT+CUSD** 命令。无论上层 Framework 走 AIDL 还是 HIDL 路径，RIL Daemon 层的 AT 命令交互保持不变：

```
AT+CUSD=1,"*101#",15

// 参数说明：
// 第一个参数：1 = 启用 USSD，2 = 取消 USSD 会话
// 第二个参数：USSD 字符串内容
// 第三个参数：编码方式，15 = 7-bit GSM 默认字母表
```

#### 3.5.3 Modem 处理

Modem 接收到 AT 命令后：

1. 将 USSD 消息封装到 **MM（Mobility Management）** 层信令
2. 通过 **DCCH（Dedicated Control Channel）** 发送至网络
3. 网络侧的 **USSD 网关/HLR/VLR** 处理请求
4. 网络返回响应，Modem 通过 `+CUSD` 主动上报给 RIL Daemon

### 3.6 阶段六：网络响应与回调流程

#### 3.6.1 网络响应类型

网络侧返回的 USSD 响应通过 `UssdModeType` 枚举定义（AIDL HAL），与旧版数字常量对应关系如下：

| `UssdModeType` 枚举 | 数值 | 含义 | 说明 |
|---------------------|------|------|------|
| NOTIFY | 0 | USSD-Notify | 网络主动推送或会话结束（无需进一步操作） |
| REQUEST | 1 | USSD-Request | 需要进一步操作，等待用户输入 |
| NW_RELEASE | 2 | 会话终止 | 网络端终止 USSD 会话 |
| LOCAL_CLIENT | 3 | 另一客户端响应 | 由其他实体（如 SIM Toolkit）处理 |
| NOT_SUPPORTED | 4 | 操作不支持 | 网络不支持该 USSD 代码 |
| NW_TIMEOUT | 5 | 网络超时 | 网络响应超时 |

> **注意**：旧版 `CommandsInterface` 仅定义了 `USSD_MODE_NOTIFY(0)`、`USSD_MODE_REQUEST(1)`、`USSD_MODE_NW_RELEASE(2)` 三个常量，而 AIDL `UssdModeType` 枚举扩展到了 `NW_TIMEOUT(5)`。

#### 3.6.2 Modem 主动上报

网络响应通过 **主动结果码（Unsolicited Result Code）** 由 Modem 上报：

```
+CUSD: <m>[,<str>,<dcs>]

// 示例：
+CUSD: 1,"Your balance is 50.00 CNY",15  // 需要进一步操作
+CUSD: 0,"Operation completed",72        // 会话结束
```

#### 3.6.3 RIL Daemon 上报至 Framework

RIL Daemon 将 `+CUSD` 主动上报转换为 HAL 层的 `onUssd` indication。根据 HAL 版本不同，分别由 AIDL 或 HIDL 接口接收。

#### 3.6.4 AIDL 接收路径（VoiceIndication）

这是当前 AOSP 主推的接收路径。`VoiceIndication` 实现 `IRadioVoiceIndication.aidl` 接口，直接处理 USSD 下行消息：

```java
// frameworks/opt/telephony/.../VoiceIndication.java
public class VoiceIndication extends IRadioVoiceIndication.Stub {
    private final RIL mRil;

    public VoiceIndication(RIL ril) {
        mRil = ril;
    }

    /**
     * Indicates when a new USSD message is received. The USSD session is assumed to persist if the
     * type code is REQUEST, otherwise the current session (if any) is assumed to have terminated.
     */
    public void onUssd(int indicationType, int ussdModeType, String msg) {
        mRil.processIndication(HAL_SERVICE_VOICE, indicationType);

        if (mRil.isLogOrTrace()) mRil.unsljLogMore(RIL_UNSOL_ON_USSD, "" + ussdModeType);

        // 构造 String[] 响应，与上层 GsmCdmaPhone 约定格式一致
        String[] resp = new String[]{"" + ussdModeType, msg};
        if (mRil.mUSSDRegistrant != null) {
            mRil.mUSSDRegistrant.notifyRegistrant(new AsyncResult(null, resp, null));
        }
    }
}
```

> **绑定关系**：`RIL.java` 构造函数中执行 `mVoiceIndication = new VoiceIndication(this)`，并通过 `((RadioVoiceProxy) serviceProxy).getAidl().setResponseFunctions(mVoiceResponse, mVoiceIndication)` 将其注册为 AIDL Voice Service 的 indication callback。

#### 3.6.5 HIDL 兼容路径（RadioIndication）

对于旧版 HIDL HAL，`RadioIndication` 同样实现了 `onUssd()`：

```java
// frameworks/opt/telephony/.../RadioIndication.java
public void onUssd(int indicationType, int ussdModeType, String msg) {
    mRil.processIndication(HAL_SERVICE_RADIO, indicationType);

    if (mRil.isLogOrTrace()) mRil.unsljLogMore(RIL_UNSOL_ON_USSD, "" + ussdModeType);

    String[] resp = new String[2];
    resp[0] = "" + ussdModeType;
    resp[1] = msg;
    if (mRil.mUSSDRegistrant != null) {
        mRil.mUSSDRegistrant.notifyRegistrant(new AsyncResult(null, resp, null));
    }
}
```

> **差异点**：HIDL 路径使用 `HAL_SERVICE_RADIO`，而 AIDL 路径使用 `HAL_SERVICE_VOICE`，这是 HAL 服务拆分后的新架构。

#### 3.6.6 上层接收链路（GsmCdmaPhone）

`mUSSDRegistrant` 的接收端在 `GsmCdmaPhone`：

```java
// frameworks/opt/telephony/.../GsmCdmaPhone.java
// 构造函数中注册
mCi.setOnUSSD(this, EVENT_USSD, null);

// handleMessage 中处理
 case EVENT_USSD:
    ar = (AsyncResult) msg.obj;
    String[] ussdResult = (String[]) ar.result;
    if (ussdResult.length > 1) {
        try {
            onIncomingUSSD(Integer.parseInt(ussdResult[0]), ussdResult[1]);
        } catch (NumberFormatException e) {
            Rlog.w(LOG_TAG, "error parsing USSD");
        }
    }
    break;
```

```java
// GsmCdmaPhone.java#onIncomingUSSD
private void onIncomingUSSD(int ussdMode, String ussdMessage) {
    boolean isUssdRequest = (ussdMode == CommandsInterface.USSD_MODE_REQUEST);
    boolean isUssdError = (ussdMode != CommandsInterface.USSD_MODE_NOTIFY
            && ussdMode != CommandsInterface.USSD_MODE_REQUEST);
    boolean isUssdRelease = (ussdMode == CommandsInterface.USSD_MODE_NW_RELEASE);

    // 匹配 pending USSD
    GsmMmiCode found = null;
    for (int i = 0, s = mPendingMMIs.size(); i < s; i++) {
        if (((GsmMmiCode) mPendingMMIs.get(i)).isPendingUSSD()) {
            found = (GsmMmiCode) mPendingMMIs.get(i);
            break;
        }
    }

    if (found != null) {
        if (isUssdRelease) {
            found.onUssdRelease();
        } else if (isUssdError) {
            found.onUssdFinishedError();
        } else {
            found.onUssdFinished(ussdMessage, isUssdRequest);
        }
    } else if (!isUssdError && !TextUtils.isEmpty(ussdMessage)) {
        // 网络发起的 USSD（非用户主动请求）
        GsmMmiCode mmi = GsmMmiCode.newNetworkInitiatedUssd(
                ussdMessage, isUssdRequest, GsmCdmaPhone.this, mUiccApplication.get());
        onNetworkInitiatedUssd(mmi);
    }
}
```

### 3.7 阶段七：UI 层结果呈现

#### 3.7.1 GsmMmiCode 处理响应

```java
// frameworks/opt/telephony/.../GsmMmiCode.java
@Override
public void handleMessage(Message msg) {
    switch (msg.what) {
        case EVENT_USSD_COMPLETE:
            AsyncResult ar = (AsyncResult) msg.obj;
            if (ar.exception != null) {
                mState = State.FAILED;
                mMessage = getErrorMessage(ar);
            } else {
                mState = State.COMPLETE;
                mMessage = (String) ar.result;
            }
            // 通知 UI 更新
            notifyMmiComplete();
            break;
            
        case EVENT_USSD_CANCEL_COMPLETE:
            // 取消 USSD 会话响应
            break;
    }
}
```

#### 3.7.2 PhoneGlobals 接收 MMI 完成事件

当 `GsmMmiCode` 状态变更（`COMPLETE`/`FAILED`/`CANCELLED`）时，会调用 `onMMIDone()` 通知 `Phone`：

```java
// frameworks/opt/telephony/.../GsmMmiCode.java
void onMMIDone() {
    if (mPhone != null) {
        mPhone.onMMIDone(this);
    }
}
```

`Phone`（`GsmCdmaPhone`）将事件转发给所有注册的 `MmiCompleteRegistrant`：

```java
// frameworks/opt/telephony/.../Phone.java
public void onMMIDone(MmiCode mmi) {
    mMmiCompleteRegistrants.notifyResult(mmi);
}
```

在 Phone 应用（`PhoneGlobals`）中，已在初始化时注册了对 `MMI_COMPLETE` 事件的监听：

```java
// packages/services/Telephony/.../PhoneGlobals.java
mCM.registerForMmiComplete(mHandler, MMI_COMPLETE, null);

// Handler 处理
 case MMI_COMPLETE:
    onMMIComplete((AsyncResult) msg.obj);
    break;

private void onMMIComplete(AsyncResult r) {
    MmiCode mmiCode = (MmiCode) r.result;
    final Message message = Message.obtain(mHandler, MMI_CANCEL);
    PhoneUtils.displayMMIComplete(mmiCode.getPhone(), getInstance(),
            mmiCode, message, null);
}
```

#### 3.7.3 PhoneUtils.displayMMIComplete 显示对话框

`PhoneUtils.displayMMIComplete()` 是显示 USSD/MMI 结果的核心方法，它会根据 `MmiCode.State` 创建并展示系统级对话框：

```java
// packages/services/Telephony/.../PhoneUtils.java
public static void displayMMIComplete(final Phone phone, Context context,
        final MmiCode mmiCode, Message dismissCallbackMessage,
        AlertDialog previousAlert) {

    MmiCode.State state = mmiCode.getState();
    String text = null;
    int title = 0;

    switch (state) {
        case FAILED:
            title = com.android.internal.R.string.mmiError;
            text = mmiCode.getMessage();
            break;
        case PENDING:
            // USSD-Request 状态，等待用户输入
            title = com.android.internal.R.string.ussdRunning;
            text = mmiCode.getMessage();
            break;
        case COMPLETE:
            title = com.android.internal.R.string.mmiComplete;
            text = mmiCode.getMessage();
            break;
        case CANCELLED:
            title = com.android.internal.R.string.mmiCancelled;
            text = mmiCode.getMessage();
            break;
    }

    // 创建 AlertDialog 并设置为系统对话框类型
    AlertDialog newDialog = new AlertDialog.Builder(context)
            .setTitle(title)
            .setMessage(text)
            .setPositiveButton(android.R.string.ok, null)
            .setCancelable(true)
            .create();

    newDialog.getWindow().setType(WindowManager.LayoutParams.TYPE_SYSTEM_DIALOG);
    newDialog.show();
}
```

> **注意**：对话框类型为 `TYPE_SYSTEM_DIALOG`，即使当前不在 Phone 应用界面，USSD 结果也能以系统对话框形式悬浮展示。

#### 3.7.4 MMIDialogActivity 启动流程

对于某些特定场景（如需要从 Activity 上下文管理生命周期），系统会启动 `MMIDialogActivity` 来承载 MMI 交互：

```java
// packages/services/Telephony/.../MMIDialogActivity.java
public class MMIDialogActivity extends Activity {
    private Handler mHandler;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        mHandler = new Handler() {
            @Override
            public void handleMessage(Message msg) {
                switch (msg.what) {
                    case PhoneGlobals.MMI_COMPLETE:
                        onMMIComplete((AsyncResult) msg.obj);
                        break;
                }
            }
        };
        mPhone.registerForMmiComplete(mHandler, PhoneGlobals.MMI_COMPLETE, null);
    }

    private void onMMIComplete(AsyncResult r) {
        MmiCode mmiCode = (MmiCode) r.result;
        PhoneUtils.displayMMIComplete(mPhone, this, mmiCode, null, null);
        finish(); // 展示完成后结束 Activity
    }
}
```

`MMIDialogActivity` 通过以下 Intent 启动：

```java
Intent intent = new Intent(context, MMIDialogActivity.class);
intent.setFlags(Intent.FLAG_ACTIVITY_NEW_TASK | Intent.FLAG_ACTIVITY_EXCLUDE_FROM_RECENTS);
context.startActivity(intent);
```

#### 3.7.5 需要进一步交互的 USSD 会话

当网络返回 `UssdModeType.REQUEST`（需要用户输入）时，流程如下：

1. `GsmCdmaPhone.onIncomingUSSD()` 识别 `isUssdRequest = true`
2. `GsmMmiCode.onUssdFinished(ussdMessage, true)` 保持 `PENDING` 状态
3. `PhoneUtils.displayMMIComplete()` 展示带输入框的系统对话框
4. 用户输入后，调用 `GsmMmiCode.sendUssd()` 发起下一轮 `RIL_REQUEST_SEND_USSD`

完整 UI 呈现链路：

```
GsmMmiCode.onMMIDone()
    → Phone.onMMIDone()
    → mMmiCompleteRegistrants.notifyResult()
    → PhoneGlobals.onMMIComplete()
    → PhoneUtils.displayMMIComplete()
    → AlertDialog (TYPE_SYSTEM_DIALOG) 展示结果

// 或 Activity 上下文路径
    → MMIDialogActivity.onMMIComplete()
    → PhoneUtils.displayMMIComplete()
    → AlertDialog 展示结果
```

## 6 USSD 取消与会话管理

### 6.1 取消 USSD 会话

用户可通过以下方式取消进行中的 USSD 会话：

```java
// 发送取消请求
mPhone.mCi.cancelPendingUssd(obtainMessage(EVENT_USSD_CANCEL_COMPLETE));

// 对应 AT 命令
AT+CUSD=2  // 终止当前 USSD 会话
```

### 6.2 会话状态机

`GsmMmiCode` 内部维护 USSD 会话状态：

```
PENDING ──► CANCELLED  (用户取消)
   │
   ├── 收到 UssdModeType.REQUEST (1) ──► 保持 PENDING，等待用户输入
   │
   ▼
COMPLETE ◄── 收到 UssdModeType.NOTIFY (0) 或 NW_RELEASE (2)
   │
   ▼
FAILED ◄──── 网络错误 (NOT_SUPPORTED/LOCAL_CLIENT/NW_TIMEOUT)
```

> **注意**：当 `onUssdFinished(ussdMessage, isUssdRequest=true)` 被调用时（即 `UssdModeType.REQUEST`），`GsmMmiCode` 保持 `PENDING` 状态，不调用 `mPhone.onMMIDone()`，因此 UI 层会保留输入对话框等待用户继续交互。只有当收到 `NOTIFY(0)` 或 `NW_RELEASE(2)` 时，状态才迁移至 `COMPLETE`。

## 7 IMS 下的 USSD 处理

在 **VoLTE/IMS** 网络环境下，USSD 可能通过 IMS 信令（SIP MESSAGE）传输，而非传统电路域。此时流程略有不同：

```
Phone → ImsPhone → ImsPhoneCallTracker → ImsService → ImsMmTelManager
                                            ↓
                                     IMS Modem (SIP MESSAGE)
                                            ↓
                                       IMS 核心网
```

IMS USSD 使用 SIP MESSAGE 方法承载 USSD 内容，响应通过 SIP 200 OK 或 MESSAGE 返回。

## 8 常见问题与调试

### 8.1 常见故障

| 现象 | 可能原因 | 排查方向 |
|------|---------|---------|
| 提示 "MMI 已完成" | USSD 代码格式错误或不支持 | 确认代码有效性，检查 log 中 `+CUSD: 4` |
| 提示 "连接问题或无效 MMI 代码" | RIL/Modem 未正确响应 | 检查 RIL log，确认 AT+CUSD 返回 |
| USSD 长时间无响应 | 网络拥塞或超时 | 检查 `+CUSD: 5`（网络超时） |
| 弹出 "USSD 正在运行" | 会话未正确关闭 | 确认取消流程是否执行 |

### 8.2 调试 Log 抓取

```bash
# 查看 RIL 相关日志（AIDL + HIDL 通用）
adb logcat -b all -s RILJ:D RILD:D AT:D *:S

# AIDL 路径专项日志（重点查看 VoiceIndication.onUssd 触发）
adb logcat -s RILJ:D VoiceIndication:D GsmMmiCode:D GsmCdmaPhone:D *:S

# 查看 Telephony 框架日志
adb logcat -s Telephony:D GsmMmiCode:D Phone:D *:S

# 抓取 radio 缓冲区日志
adb logcat -b radio
```

### 8.3 关键 Log 标签

- `RILJ`：RIL Java 层请求/响应
- `VoiceIndication`：AIDL HAL 层 USSD indication 处理（含 `onUssd` 触发日志）
- `RadioIndication`：HIDL HAL 层 USSD indication 处理
- `RILD`：RIL Daemon 层日志
- `GsmMmiCode`：MMI/USSD 解析与状态机处理
- `GsmCdmaPhone`：USSD 事件分发与 `onIncomingUSSD` 处理
- `AT`：AT 命令交互日志
- `Telephony`：Telephony 框架核心日志

## 9 总结

Android USSD 流程是一个涉及 **应用层、Framework 服务层、RIL 通信层及 Modem 硬件层** 的端到端交互过程。其核心要点包括：

1. **输入解析**：拨号应用通过 `SpecialCharSequenceMgr` 区分 Secret Code 与 USSD 代码
2. **MMI 处理**：`GsmMmiCode` 负责解析 MMI 格式并管理 USSD 会话状态机
3. **RIL 发送**：`RIL.java` 通过 `RadioVoiceProxy` 统一封装 AIDL/HIDL 双路径，将请求下发至 RIL Daemon
4. **AT 转换**：RIL Daemon 将 `RIL_REQUEST_SEND_USSD` 转换为 `AT+CUSD` 命令
5. **网络交互**：Modem 通过信令通道与网络进行会话式数据交换
6. **AIDL 回调**：网络响应由 `VoiceIndication.onUssd()`（AIDL）或 `RadioIndication.onUssd()`（HIDL）接收，通过 `mUSSDRegistrant` 投递至 `GsmCdmaPhone` 处理
7. **UI 展示**：`GsmCdmaPhone` → `Phone.onMMIDone()` → `PhoneGlobals.onMMIComplete()` → `PhoneUtils.displayMMIComplete()` 展示 `TYPE_SYSTEM_DIALOG` 系统对话框，或由 `MMIDialogActivity` 承载交互

理解这一流程对于排查 USSD 相关故障、定制运营商特性以及进行 Telephony 框架二次开发具有重要意义。
